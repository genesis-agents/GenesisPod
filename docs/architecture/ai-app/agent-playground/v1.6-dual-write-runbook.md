## Phase 1 — PR-10 切读源（T+14d，预计 2026-05-21）

### 前置条件（必须全部 ✅ 才能 merge）

- [ ] 14 天内**新建的 mission**（status=completed）100% 在 `chapters` 表有数据 — 检查 SQL：
  ```sql
  SELECT
    m.id,
    m.completed_at,
    (SELECT COUNT(*) FROM agent_playground_chapters c WHERE c.mission_id = m.id) AS new_table_count,
    (SELECT COUNT(*) FROM agent_playground_chapter_drafts d WHERE d.mission_id = m.id AND d.status = 'final') AS old_table_count
  FROM agent_playground_missions m
  WHERE m.completed_at > now() - interval '14 days'
    AND m.status = 'completed'
  ORDER BY m.completed_at DESC;
  ```
  期望：每行 `new_table_count = old_table_count`，0 容忍偏差。
- [ ] `chapter:sub-section:completed` 事件无 zod schema drop（grep DomainEventBus warn 日志）
- [ ] 任意 deep/professional mission 的 `chapter_figures` 行数 ≥ chapters 行数 × 该档 figPerCh
- [ ] 历史 mission backfill SQL（见下文 §"老 mission backfill"）已跑完，或确认接受"backfill 之前的 mission rerun 仍走 chapter_drafts fallback"

### 切换动作（PR-10 实质内容）

两处 `agentPlaygroundChapterDraft.findMany` 改为优先读新表 fallback 旧表：

```diff
// 1. backend/src/modules/ai-app/agent-playground/services/mission/rerun/ctx-hydrator.service.ts:229
- const cdRows = await this.prisma.agentPlaygroundChapterDraft.findMany({
-   where: { missionId },
- });
+ // PR-10 切读源：优先新表 → 空才 fallback 旧表（老 mission 兼容）
+ const newRows = await this.missionStore.loadPublishedChapters(missionId, userId);
+ const cdRows = newRows.length > 0
+   ? newRows.map(r => ({ ...r, attempts: 1, critique: null }))  // shape 对齐
+   : await this.prisma.agentPlaygroundChapterDraft.findMany({ where: { missionId } });

// 2. backend/src/modules/ai-app/agent-playground/services/mission/rerun/stage-rerun.dispatcher.ts:790
// 同上 if-else 模式
```

注意 shape 对齐：旧表有 `attempts` / `critique` 字段，新表没有；调用方读取这两个字段时需补 `?? 1` / `?? null`。

### 监控（merge 后 24h 紧盯）

- `agent-playground.mission:rerun-completed` rate（应保持 baseline，不掉）
- `agent-playground.chapter:writing:started` rate（rerun-incremental 触发量）
- 错误日志：`loadPublishedChapters failed` 频率 vs 前一天

### 回退（出问题时）

PR-10 是单文件 if-else 切换，回退 = revert 该 commit。**不影响数据**：

- `chapters` 表数据保留（PR-11 之前 dual-write 仍在）
- `chapter_drafts` 表数据保留
- 用户体验：rerun 走老路径，无异常

## Phase 2 — PR-11 关 dual-write（T+28d，预计 2026-06-04）

### 前置条件

- [ ] PR-10 merge 后 14 天，无 P0 incident
- [ ] `chapters` 表读取错误率 < 0.1%
- [ ] 监控仪表盘上"老 mission fallback 旧表"的命中率趋近 0（极少有用户重跑超过 28 天前的 mission）

### 切换动作

```diff
// backend/src/modules/ai-app/agent-playground/services/mission/lifecycle/mission-store.service.ts
async saveChapterDraft(...) {
- // dual-write: 旧表
- await this.prisma.agentPlaygroundChapterDraft.create({...});
- // 新表
- await this.upsertPublishedChapter({...});
+ // dual-write 已关闭（2026-06-04 PR-11），仅写新表
+ await this.upsertPublishedChapter({...});
}
```

### 回退

PR-11 一旦 merge，**新写入只在 chapters 表**。回退方案：

- revert PR-11 提交 → dual-write 恢复
- chapters 表 PR-11 期间的新数据**仍存在**，旧表会缺这一段（约 14 天数据 gap）
- 旧表 gap 会让"老 mission rerun"少数场景命中（PR-11 期间用户写的 mission，PR-11 revert 后 rerun 时 fallback 旧表无数据）→ ctx-hydrator fallback 失败 → 用户看到错误而不是沉默回退

**所以 PR-11 回退窗口很窄**——前 24 小时 hot-revert 安全，超过 24h 直接前修，不再 revert。

## Phase 3 — PR-12 drop 旧表（T+90d，2026-08-05+，可选）

### 是否要做？

**不做也可以**。`agent_playground_chapter_drafts` 表占用磁盘 ≈ 50-200 MB（按 1000 mission × 每 mission 50 attempts × 5 KB / row 估算），保留作为审计无成本。

### 如果决定做

```sql
-- 2026-08-05 PR-12（可选）：drop 旧表
-- 前置：确认无 prod 代码引用 agentPlaygroundChapterDraft
DROP TABLE agent_playground_chapter_drafts;
```

不可回滚。决定 drop 前：

1. 全仓 grep `agentPlaygroundChapterDraft\|chapter_drafts`，确认 0 引用
2. prisma schema 删除 `model AgentPlaygroundChapterDraft`
3. 数据备份（pg_dump 该表，存对象存储 30 天）

## 老 mission backfill SQL（PR-10 前可跑）

> 目的：让 PR-10 切读源后，**老 mission**（v1.6 之前创建的）rerun 时 `loadPublishedChapters` 不返空。
> 时机：PR-10 之前任意时间，推荐 T+7d（2026-05-14）给 dual-write 7 天观察期。**幂等**（基于 `(mission_id, dimension, chapter_index)` 跳过已存在）。

**脚本位置**：[`backend/prisma/migrations/manual/20260514_chapter_backfill_for_pr10.sql`](../../../../backend/prisma/migrations/manual/20260514_chapter_backfill_for_pr10.sql)

**执行方式**（NOT 自动随 prisma migrate deploy 跑）：

```bash
# 1. 备份旧表
pg_dump $DATABASE_URL --table=agent_playground_chapter_drafts \
  -f /tmp/chapter_drafts_pre_backfill_$(date +%Y%m%d).sql

# 2. 在 prod 数据库手工执行
psql $DATABASE_URL -f backend/prisma/migrations/manual/20260514_chapter_backfill_for_pr10.sql

# 3. 校验（脚本末尾自带 verification queries）
```

**Note**：

- `chapter_figures` 和 `chapter_citations` 的 backfill 更复杂（旧表无对应数据），决定**不 backfill**——老 mission rerun 时 figures/citations 由 reportArtifact JSONB 重新解析
- 用 `chapter_drafts.id` 作为新行 id（保证幂等 — 重跑相同源行不会产生 duplicate UUID）
- 多 attempts 的章节取 `MAX(attempts)` 行（terminal attempt）
- 仅 backfill `completed_at < '2026-05-07'` 的 mission（PR-8 v1.6 上线分界线）

## OpenAI 平台 spend limit

PR-13 sub-section 拼接 + DALL-E AI 生成 fallback 增加了平均每 mission 的 OpenAI 花费。**必须在 OpenAI dashboard 设 hard limit**：

1. 登录 https://platform.openai.com/settings/organization/limits
2. **Hard limit** 建议设为月度总预算的 **120%**（buffer for 异常 spike）
3. **Soft limit / email alert** 设为月度预算 **80%**（提前一周预警）
4. 验证 GenesisOpenAI 项目下设了独立 budget（防 BYOK key 与系统 key 混计）

**关键警戒值**（v1.6 上线后基于 mission 量重新校准）：

| Scale        | 单 mission OpenAI 花费估算 | 备注                        |
| ------------ | -------------------------- | --------------------------- |
| quick        | $0.10-0.30                 | 6 章 × 1 LLM call           |
| standard     | $0.50-1.50                 | 15 章 × 1 LLM call          |
| deep         | $5-12                      | 10 章 × 3 LLM call + DALL-E |
| professional | $20-40                     | 12 章 × 4 LLM call + DALL-E |

成本 spike 根因诊断：

- `chapter:sub-section:completed` 事件量 vs 预期（每 mission deep 档应 = 30 次）
- `figure-curator` 命中"AI 生成 fallback"率（应 < 30%，否则 image-search 兜底失效）
- `assertHardContract` 触发 retry-budget-allowed 的 mission 占比（每次 retry 重新跑，成本 ×2）
