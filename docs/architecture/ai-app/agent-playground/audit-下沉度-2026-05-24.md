## 六、cost-strategy 相关代码归属判断

`playground-cost-strategy-v1.md` 定义了 `deep/report/brief/insight` 四个执行形状和 `CostPlan` 数据结构。审计发现相关代码分布如下：

### 当前已归位到 ai-harness/guardrails/ 的部分（正确）

| 能力                     | 文件                                                                                                            | 状态                              |
| ------------------------ | --------------------------------------------------------------------------------------------------------------- | --------------------------------- |
| 预算池上限解析           | `ai-harness/guardrails/budget/resolved-budget-caps.ts`（via facade）                                            | 正确，playground adapter 调用     |

| 全局 maxCredits/wallTime | `resolveMissionCredits`/`resolveMissionWallTimeMs` 在 `run-mission.dto.ts`（L~80-150），调 `ResolvedBudgetCaps` | 正确，DTO 层业务解析留 playground |

### 应下沉到 ai-harness/guardrails/ 的部分（待实现）

`cost-strategy-v1.md` §5.1 定义了 `CostPlan` 类型：

```ts
type CostPlan = {
  executionMode: "brief" | "insight" | "deep" | "report";
  priorityDimensions: string[];
  coverageDimensions: string[];
  writingMode: "summary_only" | "selective_chapters" | "full_report";
  reviewMode: "sampled" | "final_only" | "full";
  reserveRatio: number;
};
```

该类型目前**尚未实现**（文档状态 Draft）。当它实现时：

- `CostPlan` 类型定义和 `S2 leader` 的 cost-plan emit 逻辑 → **应留在 playground**（playground-specific 业务决策）
- 动态 downgrade 逻辑（`report → deep → insight`，§8.3）和 retry 价值评估（`RetryDecision`，§8.1）→ **应下沉到 `ai-harness/guardrails/runtime/`**，因为 social/radar 的 agent team 也需要相同的降级机制
- 角色-模型层级映射（§7）→ **应留在 playground** 的 `playground-tuning-profile.ts`（已存在，playground 专属模型策略）
- 证据缓存策略（§9）→ **下沉到 `ai-harness/guardrails/`** 的缓存层（budget+缓存联动属通用能力）

### 当前 playground 保留的预算逻辑（正确定位）

| 文件                                             | 内容                                        | 归属判断                                 |
| ------------------------------------------------ | ------------------------------------------- | ---------------------------------------- |
| `run-mission.dto.ts:145`（`DEPTH_BUDGET_TIERS`） | deep/standard/quick 预算档位默认值          | 正确留 playground（playground 产品规格） |
| `api/contracts/word-budget.contract.ts`          | 报告总字数公式（depthBase × lengthProfile） | 留 playground，等 social 同需求时迁移    |

**结论**：cost-strategy 目前的代码归属是合理的，`resolved-budget-caps.ts` 在 harness 已正确实现通用预算封顶。等 `CostPlan` 实现时，注意把 runtime 降级机制放进 harness guardrails，避免每个 app 各自实现降级逻辑。
