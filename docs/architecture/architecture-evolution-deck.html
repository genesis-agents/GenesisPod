提交结构

fix : feat ≈ 2.0 : 1。修复量是新功能的两倍——高频迭代下的缺陷与返工压力清晰可见，是后文「重构债务」判断的第一个证据。

治理强度

refactor 807 + chore 497 + docs 292 ≈ 20% 提交不产出新功能，全部投向结构与规范。对一个 7 个月的项目而言异常之高。

质量基建

test 216 + perf 32 + ci 5。测试占比 2.7%——相对架构治理的投入，自动化测试是明显的洼地。

04① 创世 → ② 引擎抽象：把能力从应用里「下沉」

① 单体起步 · 2025.11

以「DeepDive Engine」之名落地：深度研究、多步骤规划报告、YouTube 评论。所有 AI 能力与业务逻辑耦合在应用模块内。

② AI Engine 抽象 · 2025.12–2026.01

关键转折：提出 capability sinking（能力下沉）——把领域无关的服务从 AI Teams 抽进 AI Engine；RAG 核心迁入；删除 ai-agents 冗余。App 变薄、Engine 变厚的主线就此确立。

TaskProfile：去硬编码的支点

2026.01 引入 TaskProfile 语义化参数（creativity/outputLength），统一所有模块经 chat() 入口调用 LLM。从此「model:"gpt-4o"/temperature:0.7」硬编码被制度性禁止——这条红线沿用至今。

数据库三合一

移除 MongoDB / Neo4j / Qdrant，统一到 PostgreSQL 16（结构化 + JSONB + 图关系）+ Redis 7，宣称成本 -70%~75%。一次果断的减法。

05③ 品牌动荡 + ④ Facade 治理：边界开始硬化

③ 四次 rebrand 的代价

  • DeepDive Engine → AI Teams Engine（01.06）
  • → Raven + flock logo（01.30）当天即 revert
  • → Genesis.ai + 松鼠 logo（02.16，108 文件清理）
  • → 最终 GenesisPod
品牌反复横跳消耗了可观精力，且多次 revert。教训沉淀：此后品牌名一律走 config.brand.*，禁止硬编码——把动荡转化成了一条规范。

④ 三层 Facade 边界（架构成熟的真正起点）

  • 强制 Facade 导入ai-app 禁止穿透 ai-engine 内部路径
  • God Facade 拆解单个 38 参数巨型 Facade → 5 个领域 Facade
  • 137 个消费方迁移一次性把全量调用切到域 Facade
  • ESLint no-restricted-importsIDE 实时 + pre-commit 拦截违规导入
这是项目第一次把「架构约定」变成「机器可执行的拦截」——奠定了后续看护体系。

06⑤ Kernel 实验 → ⑥ Harness 诞生：Agent 运行时归位

⑤ ai-kernel：一次短命的层（02–04）

引入 KernelContext / ProgressTracker / Memory 想做「内核层」，深度集成进各 App。但定位与 ai-engine 重叠、边界说不清——仅存活约两个月。

2026.04.22：删 kernel，立 harness

同一天，ai-kernel 被并入 engine/runtime 并删除，Harness 抽象层正式诞生。承认前一次分层错误、果断推倒重来。

⑥ Harness 六阶段一次成型

  • P1 Harness 抽象层骨架
  • P2 ReAct loop + tool invoker + memory bridge
  • P3 SKILL.md 系统：parser/registry/loader/activator
  • P4 subagent spawner + 3 级隔离
  • P5 上下文工程：compactor + pruner + manager
  • P6 checkpoint / resume + skill learner
明显借鉴 Claude Code 的 agent 运行时范式（SKILL.md、subagent 隔离、context 压缩）。

07⑦ MECE 大重构：17 个波次，把一切「归位」

2026.05 用 W1–W17 波次把 harness 拆成 11 个全业界标准词顶层聚合，engine 收敛为 12 个聚合。核心准则——

  • engine 不知 agent/mission无状态基元；harness 才懂 agent/mission
  • MCP 属 engine 不属 harness与 OpenAPI/function 同层的 tool 适配器
  • 同名概念全项目唯一tools 只在 engine，SkillRegistry 只 1 个
  • 顶层全用业界标准词禁造 kernel/runtime/process/governance

典型波次（搬家而非新功能）

W6 governance/* 全归位 + 删目录
W8 process/* 全归位 + 删目录
W12 dissolve runtime facade
W13 rename kernel → agents
W14 move execution into runner
W16 relocate mission lifecycle

同时发生的跨层迁移

credentials L2→L1 下沉（BYOK key 解析零 agent 状态）；planning 目录整体删除、能力迁入 harness/runtime；auto-dream、adaptive-replanner 等逐个跨层搬家。

审视：结果是干净的，但「17 个波次 + 整月搬家」也暴露了——前几次分层抽象的一次到位率偏低，演进成本被推迟到了这里集中偿还。

09⑧ 三层架构看护：把「合规」变成「可执行」

① ESLint 实时拦截

no-restricted-imports：engine 不得 import harness；app 不得穿透内部路径。IDE 实时 + lint-staged pre-commit。

② 架构 spec 测试

verify:arch：jest 整目录多套件，拦截 ESLint 漏掉的动态 import / 注释逃逸（拓扑/权限/词汇/能力多维）。

③ pre-push + CI 门

.husky/pre-push 第 0 步跑 verify:arch；CI 的 arch-boundary job 汇入合并门，失败即拒合。

真正稀缺之处：多数团队的分层只活在文档里。这里把它做成了三道机器闸门——这是本仓最值得肯定的工程资产。
但要诚实：借鉴 Claude Code 的「10 条反向洞察」目前 全部 honor-only（仅人工 review）；85% 覆盖率阈值也未真正进 CI。「9.8/10 合规」含一定自证成分。

10⑨ 终局转向：从「给人用的工具」到「自驱动的公司」

self-driven agent team · 2026.06

AI Ask 里一句话触发完整链路：plan → build → execute → deliver。自动组队、HITL 审批门、durable event journal、detached 后台调度、dead-pod 孤儿检测。

复用而非重造

关键提交：refactor(self-driven): reuse engine/harness, drop hand-rolled election + approval——并加 test(arch) 守护它不再重新发明 engine/harness 能力。前期架构投资在这里开始兑现。

产品野心的上移轨迹

研究工具 → AI 应用平台 → Agent 运行时（Harness）→ 单人公司 OS（One-Person-Company）。每一层都建立在前一层的架构基建之上。

健康信号

6 月提交回落到 453（5 月的 ~1/5），重心从「搬家」明确转回「业务」（self-driven / org-chart / mission gallery）。架构进入相对稳定期。

11专业审视 · 做对的 vs 欠下的

✓ 真正的工程资产

  • 意图明确的分层 + MECE 准则engine 无 agent 状态、概念全项目唯一，边界讲得清
  • 三层自动化看护把架构合规从「靠自觉」变成「可执行」，稀缺
  • 能力下沉主线稳定App 薄 / Engine·Harness 厚，self-driven 验证了复用红利
  • 果断的减法三库合一(-70%)、删 kernel/intent-gateway 空壳
  • 去硬编码红线TaskProfile + 空字符串 fallback，沿用至今未破
  • 向 SOTA 学习系统性借鉴 Claude Code 运行时与 10 条反向洞察

✗ 待偿的债务

  • 重构/返工占比过高fix:feat≈2:1；refactor 807；kernel 建了又删、planning 整删、17 波次——抽象一次到位率低
  • 五月搬家潮单月 26% 提交是纯目录归位，业务产出比偏低
  • 分层认知负荷L1–L5+L2.5；engine↔harness 边界需整篇文档+spec 才守得住
  • 测试洼地test 仅 2.7%，与架构治理投入严重不匹配
  • 护栏含自证10 条反向洞察全 honor-only；覆盖率阈值未进 CI
  • 文档驱动的脆弱性数千行 CLAUDE.md + 大量 redesign 文档，与代码同步成本高、新人陡峭

12研判与建议 · 从「搬家」转向「沉淀」

1 · 冻结结构，进入维护期

分层已趋稳、6 月重构回落——明确宣布结构冻结，把 W1–W17 式的大搬家划上句号。再动结构需走 ADR + 收益论证。

2 · 把 honor 护栏升级为 spec

10 条反向洞察里的高频项（stop_reason 误判、prompt cache 破坏、跨线程状态污染）逐条加单测/spec；把 85% 覆盖率阈值真正接入 CI。让「9.8/10」名副其实。

3 · 补测试洼地

2.7% 的测试占比撑不住「自驱动公司」这种高自治系统。优先为 self-driven 链路（plan→build→execute）和 harness runner 建集成测试网。

4 · 沉淀演进 ADR，止住反复

kernel/品牌/分层都出现过「建了又删/改了又回退」。建立轻量 ADR 记录每次架构决策的为何，让下一次抽象一次到位，而非推迟到下一轮波次偿还。

一句话总结：这是一个架构意识远超同龄项目、但为分层付出了高额学费的代码库。基建已成，红利初现（self-driven）——接下来比拼的不是「再分一层」，而是把已有结构守稳、把测试补齐、把决策沉淀。
