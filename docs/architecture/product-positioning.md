# GenesisPod 产品定位与方案文档

> 2026-06-09 落档。本文是 GenesisPod「一人公司 OS」的**权威定位北极星**，统领产品方向、护城河、术语体系、信息架构与落地分期。
>
> **与既有文档关系**：
>
> - 取代 [opc-hero-model.md](./opc-hero-model.md) 中的「英雄」术语体系（实现机制仍以该文为准，但**对外呈现一律改用「专家 Agent」**，见 §5 术语迁移）。
> - 上层定位依据，下游所有 company/marketplace 的呈现改动以本文 §5/§6 为准。

## 1. 一句话定位

> **GenesisPod = 一人公司 / 专业个体的「专家 Agent」平台**：用户不组装零件、不养通用 agent，而是**雇佣开箱即用、分工有审校、有交付标准的专家 Agent**，派任务即拿回**可直接交付的专业业务成果**。
>
> **「专家 Agent」命名**：「专家」承载专业判断力（护城河、2B 可信的胜任感），「Agent」承载自主智能体的智能感。统一写法「专家 Agent」（中间留空格）；纯中文场景兜底「专家智能体」。实例命名用职能头衔（深度研究专家 / 尽调分析师 / 行业顾问），不用雅称。

面向 **2B（极小团队）/ 2 专业个体**，不面向开发者，不面向「管理个人设备/生活」的消费场景。

---

## 2. 定位坐标（我们在哪，左右是谁）

| 维度     | 零件市场（MCP / 技能库） | **GenesisPod（专家 Agent）**          | 通用自主 Agent（OpenClaw/龙虾、Lindy） |
| -------- | ------------------------ | ------------------------------------- | -------------------------------------- |
| 交易单位 | 工具 / 技能 / prompt     | **干完的活（成品交付）**              | 一只通用 agent                         |
| 目标用户 | 开发者（会组装）         | **一人公司老板 / 专业个体（要结果）** | 个人（养它干杂活）                     |
| 形态     | 自己拼装                 | **开箱即用、自动成军、有验收**        | 自己养、授予设备 root                  |
| 核心强项 | 灵活、可组合             | **可信、一致、可问责**                | 全能、灵活、生态势头猛                 |
| 致命软肋 | 要求用户会拼             | 灵活性弱、垂直                        | 不可控、会闯祸、质量抽卡               |

**战略结论**：不在两端任何一端硬刚。左端用户画像不符（我们的用户不是开发者）；右端打不过开源 + 免费 + 势头。我们占据中间**唯一别人做不好的位置：专业交付的可信度**。

参考事实（2026-03，OpenClaw/「龙虾」）：开源个人 agent，GitHub 25 万星超过 React；但已出现账户盗刷、电脑被远控、文件被一键删除等事故，工信部发布安全风险预警（「龙虾悖论」：越强越危险）。**它最大的软肋＝我们的结构性强项。**

## 3. 目标用户与待办任务（JTBD）

- **谁**：一人公司 / 极小团队 / 专业个体（咨询、投研、律所、MCN、独立顾问、自由职业）。
- **正向 JTBD**：「我要一份**能交给客户/老板**的专业成果（尽调、研报、合规分析、方案），质量稳定、我不用懂 AI、出问题能复盘。」
- **反向 JTBD（明确不做）**：「整理我的桌面文件 / 控制我的电脑 / 替我发消息」——这是通用龙虾的活，**主动放弃**。

## 4. 护城河四层（★＝最防御层，含代码现状）

专家 Agent 的价值 = **被编码的专业判断**，沉淀在四层。前三层已真实建成，第四层缺失，是当前 P0。

| 层                | 编码的判断                    | 承载点（代码）                                           | 现状                                                                  |
| ----------------- | ----------------------------- | -------------------------------------------------------- | --------------------------------------------------------------------- |
| 工作流程（打法）  | 这类活分几步、先后、何时回头  | `MissionPipelineRegistry` / capability steps             | ✅ 真跑（deep-insight: plan/research/reconcile/analyze/write/review） |
| 团队配置（阵型）  | 配哪些角色、谁主谁辅谁审      | capability roles + `tryRunRealAgent` 真叶子 agent        | ✅ 真成军（researcher 带真 web-search/ReAct）                         |
| 技能              | 每步注入的领域方法论正文      | `buildSkillInstructions`（注入 SKILL.md 正文）           | ✅ 真生效（非 id 字符串）                                             |
| **★ 验收 rubric** | **什么叫做对了 + 不达标重跑** | 应在 `runViaCapability` 终态前 / `ai-harness/evaluation` | ❌ **缺失（P0）**                                                     |

**关键诚实结论**：当前 `runReview`（`company-mission.service.ts` 中）只是 **Leader 用同一个 LLM 自我总结 + 自评**，没有独立验收、没有合格门槛、没有不达标重跑；DTO 中无 `rubric` 字段，贡献者无法声明「什么叫做对了」。即——**前三层让我们比龙虾稳、比龙虾安全，但「懂什么是好」这一层尚未上路**。而这一层正是模型再强也替代不了的命门（人的标准与品味，非模型能力），是对抗龙虾最锋利的一刀。

## 5. 术语迁移：去「英雄」，回归 2B / 专业视角

**原则**：去游戏化（英雄 / 雅称 / 收下 / 派单），回归专业、可信赖的 2B 用语。核心单位统一为「**专家 Agent**」（中间留空格；纯中文场景兜底「专家智能体」），与「一人公司 / 团队 / CEO」既有框架自洽。「专家」承载专业判断力，「Agent」承载智能感——但对外不把「智能」当卖点喊，让「专家」承重（避免落回与通用 agent 拼 raw 智能的战场）。

### 5.1 词表映射（对外呈现层，逐词替换）

| 现（英雄 / 游戏化）     | 新（专家 Agent / 2B）                      | 备注                |
| ----------------------- | ------------------------------------------ | ------------------- |
| 英雄                    | **专家 Agent**（纯中文兜底「专家智能体」） | 核心单位            |
| 英雄市场                | **专家市场**                               | marketplace         |
| 收下英雄                | **录用** / 雇佣                            | adopt CTA           |
| 我的英雄                | **我的团队**                               | /agents 列表        |
| 英雄任务 / 派单         | **任务** / 指派任务                        | /missions           |
| 雅称（知微/洞玄/明察…） | **去掉** → 用职能名                        | 见 §5.2             |
| 自带打法                | **标准工作流程**                           | 卡片 section        |
| 阵型                    | **协作分工** / 团队配置                    | 卡片 section        |
| 一句话人设（tagline）   | **职能简介**                               | cosmetic 字段语义改 |

### 5.2 实例命名：删掉雅称池，改用职能名（用户明确要求）

- **现状**：`company-hero.service.ts` 的 `HERO_NAMES`（知微/洞玄/明察…）按用户现有数循环取雅称。
- **改为**：实例名 = **能力 manifest 的职能标题**（如「深度研究专家」）；同职能多个时加序号后缀（「深度研究专家 2」）。
- 删除 `HERO_NAMES` 常量；`adoptHero` 命名逻辑改为查 manifest 标题 + 去重。
- 用户仍可手动改名（保留 `updateHero.name`）。

> 说明：这是「对外呈现 + 实例命名」层的改动。数据模型 `CompanyHero` / API 路径 `/company/heroes` 等**内部标识符可暂不改名**（避免大范围迁移成本），仅 UI 文案与实例显示名按本表替换；后续如要彻底去 hero 内部命名，单列一个重构波次。

### 5.3 改动范围（前端，约 8 个文件）

`MarketplaceView.tsx`、`HeroRosterView.tsx`、`MissionGalleryView.tsx`、`MissionRunView.tsx`、`TalentPoolView.tsx`、`AgentTeamSection.tsx`、`app/agents/page.tsx`、`nav-config.ts`（+ i18n 文案）。后端仅 `company-hero.service.ts` 的命名逻辑。

## 6. 信息架构 / 左侧菜单调整方案

### 6.1 设计主轴

专家 Agent 的核心使用闭环：**招（专家市场）→ 管（我的团队）→ 用（任务/交付）**。侧栏 IA 应让这条主轴清晰。

### 6.2 当前侧栏（`nav-config.ts`）

```
(顶) AI 问答
AI 广场:    发现 / 英雄市场 / 体验场
我的工作台:  我的资料库 / 我的英雄 / 英雄任务
深度洞察:    radar / insights / research
(隐藏) 推演决策 / 内容工坊
```

### 6.3 最终方案（用户 2026-06-09 拍板：合并双 Tab + 探索改名挪组）

最终侧栏分组（重心重排：用户自己的「我的工作台」提到最前，获取/实验类「AI 广场」沉到最后）：

```
(顶)  AI 问答
我的工作台:   我的前沿(/explore) · 我的知识库(/library) · 我的团队(/agents, 双Tab)
深度洞察:     AI 雷达 · 话题洞察 · 深度研究  (radar / insights / research)
发现能力:     专家市场(/marketplace) · AI 实验场(/agent-playground)   ← 挪到最后，自成一组
(隐藏)  推演决策 / 内容工坊
```

逐条改动：

| 条目                | 现                                | 改为                                                                                 | 说明                                                                    |
| ------------------- | --------------------------------- | ------------------------------------------------------------------------------------ | ----------------------------------------------------------------------- |
| /explore            | 发现 / AI 探索（AI 广场，Search） | **我的前沿**（移入「我的工作台」首位）                                               | 重定位「前沿信息感知」；图标 Search → **Telescope**（备选名：我的情报） |
| /library            | 我的资料库                        | **我的知识库**                                                                       | 仅文案；BookOpen 不变                                                   |
| /agents + /missions | 我的英雄 + 英雄任务（两条）       | **我的团队**（合并一条，内含双 Tab）                                                 | 图标 Crown → **Users**；删 /missions 独立 nav 条目                      |
| /marketplace        | 英雄市场                          | **专家市场**                                                                         | 仅文案；图标 Store 不变                                                 |
| /agent-playground   | 体验场                            | **AI 实验场**                                                                        | 仅文案                                                                  |
| 分组顺序            | AI广场→我的工作台→深度洞察        | 我的工作台 → 深度洞察 → **发现能力（原「AI 广场」，含 专家市场 + AI 实验场，置底）** | 重心前移 + 组改名                                                       |

> 「我的工作台」合并后原只剩 2 条（知识库 + 团队），/explore 改名挪入后回到 **3 条**；原「AI 广场」移出 /explore 后改名「**发现能力**」，剩 专家市场 + AI 实验场，整组下沉至底部。

**「我的团队」页内双 Tab（用 canonical `components/ui/tabs/Tabs.tsx`，非自写）**：

## 7. 方案支撑（复用现有地基 + 补缺口）

**已有地基（直接复用，禁止重建）**：`CapabilityRegistry` + `MissionPipelineRegistry`（能力源）、`runViaCapability`（统一 run/persist/event 机器）、`buildSkillInstructions`（技能真注入）、`MarketplaceCatalogService`（货架投影）、BYOK + TaskProfile（模型层）。

**要补的缺口（按优先级 + 可验证目标）**：

| #   | 任务                                                                                                                                                                                           | 验证标准（verify）                                                    | 优先级 |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | ------ |
| 1   | **验收 rubric 上运行路径** —— 能力 manifest 加 `rubric`；`runViaCapability` 终态前跑独立 evaluation（`ai-harness/evaluation` 的 critique/verify，非 leader 自评），不达标阶段重跑（封顶 N 次） | 跑一个故意残缺的 mission → rubric 不通过 → 自动重跑 → 通过才落 `done` | **P0** |
| 2   | **专家 Agent契约（Manifest）+ 强制 rubric/样例** —— 贡献流程缺 rubric 或样例交付 → 不准上架                                                                                                    | 提交无 rubric 的员工 → 被拒上架                                       | **P0** |
| 3   | **术语去英雄化 + 删雅称池**（§5）                                                                                                                                                              | UI 全量无「英雄/收下/雅称」字样；实例名 = 职能名                      | **P1** |
| 4   | **侧栏调整**（§6.3）                                                                                                                                                                           | 侧栏显示「专家市场/我的团队/我的任务」                                | **P1** |
| 5   | **市场卡片展示样例交付** —— 录用前先看货                                                                                                                                                       | 卡片可点开看一份真实成品                                              | **P1** |

## 8. 边界（写死，防止飘回必输的仗）

- ❌ 不做「更安全的龙虾」（通用设备控制 / 个人助理）——它的主场，正面输。
- ❌ 不做开发者零件市场——用户画像不符。
- ✅ 只做**需要可信、可复用、有验收标准的专业业务交付**。

---

## 9. 变现（单位 = 干完的活，三方同层自洽）

- **贡献者交什么**：一份专家 Agent Manifest = 工作流程 + 协作分工 + 技能 + **强制 rubric** + 样例交付。
- **用户买什么**：开箱即用的员工 + 可信的成品交付。
- **怎么收费**：按 mission 计费（模型成本 BYOK 甩给用户，平台收**编排 + 验收**溢价）+ 作者分成（App Store 式 70/30）+ 明星员工作者定价、平台抽成。
