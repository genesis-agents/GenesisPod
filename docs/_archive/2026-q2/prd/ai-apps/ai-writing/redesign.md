## 设计原则

> **核心原则：参考 AI Teams 的展现方式，保持产品一致性**

1. **首页用卡片列表**：与 AI Teams 首页一致
2. **详情页左 Canvas + 右面板**：与 AI Teams 详情页一致
3. **Agent 协作可视化**：节点 + 连接线 + 状态色
4. **实时反馈**：脉冲动画、流式输出、typing 指示

## 1. 概述

### 1.1 背景

当前 AI Writing 模块存在严重的产品设计问题，用户满意度极低。经过深入的代码分析和用户需求理解，我们需要从零开始重新设计这个模块。

### 1.2 当前问题分析

#### 1.2.1 用户体验问题

| 问题               | 严重度 | 具体表现                                                              |
| ------------------ | ------ | --------------------------------------------------------------------- |
| **上手成本过高**   | 严重   | 用户需要理解"项目→卷→章节→Story Bible"的复杂层级结构                  |
| **概念过于技术化** | 严重   | Story Bible、一致性检查等概念对普通用户不友好                         |
| **入口点混乱**     | 中等   | 用户不知道从何开始，3种任务类型（outline/chapter/full_story）让人困惑 |
| **反馈机制缺失**   | 严重   | 任务执行后无实时反馈，用户不知道AI在做什么                            |
| **结果不可控**     | 严重   | 用户无法干预写作过程，只能被动等待                                    |

#### 1.2.2 架构设计问题

| 问题                     | 具体表现                                                                                             |
| ------------------------ | ---------------------------------------------------------------------------------------------------- |
| **过度工程化**           | 5个Agent（故事架构师、创作作家、设定守护者、一致性检查员、润色编辑）过于复杂，用户无法理解其协作方式 |
| **技术驱动而非需求驱动** | Story Bible、一致性引擎、LongContentEngine等技术概念直接暴露给用户                                   |
| **流程僵化**             | 固定的工作流（规划→上下文注入→写作→检查→编辑→审核）不适合所有写作场景                                |
| **缺乏渐进性**           | 用户无法从简单场景逐步过渡到复杂场景                                                                 |

#### 1.2.3 代码实现问题

```
当前文件结构问题：
- frontend/app/ai-writing/page.tsx (481行) - 页面过于复杂
- frontend/app/ai-writing/[id]/page.tsx (1037行) - 单文件过大
- 后端有30+个服务文件，但用户感知不到其价值
```

### 1.3 设计目标

| 目标         | 衡量标准                            |
| ------------ | ----------------------------------- |
| **简单易用** | 用户3分钟内能开始第一次写作         |
| **渐进复杂** | 基础功能0学习成本，高级功能按需发现 |
| **实时互动** | AI写作过程实时可见，用户可随时干预  |
| **结果可控** | 用户对每一步输出都有修改权          |
| **专业可信** | 利用AI Teams能力，但不暴露技术细节  |

### 1.4 非目标

- 不追求一次性生成完整小说（技术限制）
- 不替代专业写作软件（如Scrivener）的全部功能
- 不强制用户使用Story Bible等高级功能

## 2. 竞品分析

### 2.1 竞品对比

| 产品               | 定位         | 核心优势                         | 核心劣势       | 价格        |
| ------------------ | ------------ | -------------------------------- | -------------- | ----------- |
| **Notion AI**      | 工作空间集成 | 无缝集成现有工作流，上下文保持好 | 长篇写作能力弱 | $8/月       |
| **Jasper**         | 营销内容生成 | 品牌声音一致性，SEO优化          | 不适合小说创作 | $39/月起    |
| **Sudowrite**      | 小说写作助手 | 场景级写作建议，沉浸式体验       | 仅限英文       | $22/月      |
| **Novelcrafter**   | 长篇小说管理 | 角色/世界观管理，模型可选        | 学习曲线陡     | $4/月起     |
| **ChatGPT/Claude** | 通用AI对话   | 灵活，无限可能                   | 无状态，无记忆 | 免费-$20/月 |

### 2.2 最佳实践提炼

#### 2.2.1 从Notion AI学到的

```
- 不新建工具，在用户已有工作流中提供AI能力
- 用户先写，AI辅助（而非AI生成，用户审核）
- 操作即时生效，无需等待
```

#### 2.2.2 从Sudowrite学到的

```
- 场景级别（而非章节级别）的交互粒度
- "继续写"、"扩展描写"、"改变语气"等具体操作
- 保持用户在创作心流中
```

#### 2.2.3 从Novelcrafter学到的

```
- 设定管理（角色/地点/物品）对长篇创作很重要
- 但应该是可选的高级功能，而非入门门槛
- 让用户选择使用哪个AI模型
```

### 2.3 差异化机会

Genesis.ai 的独特优势：**AI Teams 多Agent协作机制**

竞品大多是单Agent模式，而我们可以：

- 让多个AI角色各司其职，但以用户不可见的方式协作
- 在后台保持一致性检查，但不打扰用户
- 提供"第二意见"功能，让不同AI视角给出建议

## 3. 产品重新设计

### 3.1 产品定位

> **AI Writing = 你的写作搭档，而非写作机器**

核心理念：

- 用户是作者，AI是协作者
- 用户决定写什么，AI帮助写得更好
- 从"一键生成"转变为"持续协作"

### 3.2 用户画像

#### 3.2.1 主要用户：创意写作者

```
- 个人作者，想写小说/故事/剧本
- 有创意和想法，但需要帮助组织和扩展
- 希望保持自己的风格和控制权
- 不是专业技术用户，不想学习复杂工具
```

#### 3.2.2 次要用户：内容创作者

```
- 需要定期产出内容（博客/公众号）
- 时间紧张，希望提高效率
- 需要保持一致的品牌声音
```

### 3.3 核心功能重新定义

#### 3.3.1 功能层级（渐进式设计）

```
┌─────────────────────────────────────────────────────────┐
│  Level 1: 即刻开始（0学习成本）                          │
│  - 直接输入，AI续写                                      │
│  - 选中文本，要求修改                                    │
│  - 无需创建项目                                          │
├─────────────────────────────────────────────────────────┤
│  Level 2: 项目管理（5分钟学习）                          │
│  - 保存为项目                                            │
│  - 管理多个作品                                          │
│  - 导出分享                                              │
├─────────────────────────────────────────────────────────┤
│  Level 3: 世界观管理（可选高级功能）                     │
│  - 角色卡片                                              │
│  - 地点设定                                              │
│  - 时间线                                                │
├─────────────────────────────────────────────────────────┤
│  Level 4: AI团队协作（专业用户）                         │
│  - 可视化Agent协作                                       │
│  - 一致性报告                                            │
│  - 多视角审稿                                            │
└─────────────────────────────────────────────────────────┘
```

## 4. 用户故事与功能需求

### 4.1 Level 1: 即刻开始

#### US-1.1 快速续写

```
作为一个创意写作者，
我想要输入开头后让AI继续写，
以便快速看到故事如何发展。

验收标准：
- [ ] 用户打开页面，直接看到写作区域
- [ ] 输入任意文本后，点击"继续写"
- [ ] AI在2秒内开始流式输出
- [ ] 用户可以随时停止
- [ ] 生成的内容自动追加到编辑器
```

#### US-1.2 选中修改

```
作为一个写作者，
我想要选中一段文字让AI改写，
以便快速优化表达。

验收标准：
- [ ] 选中文本后，出现浮动工具栏
- [ ] 提供常用操作：扩写、缩写、换种说法、更正式、更口语
- [ ] 支持自定义指令
- [ ] 修改在原位置替换或对比显示
```

#### US-1.3 AI建议

```
作为一个写作者，
我想要在任何时候获得AI的建议，
以便克服写作障碍。

验收标准：
- [ ] 随时可以向AI提问："接下来怎么写？"
- [ ] AI基于当前上下文给出3个方向建议
- [ ] 点击建议可以直接应用
```

### 4.2 Level 2: 项目管理

#### US-2.1 保存为项目

```
作为一个写作者，
我想要把写作内容保存为项目，
以便后续继续编辑。

验收标准：
- [ ] 一键"保存为项目"
- [ ] 自动提取标题（可修改）
- [ ] 项目列表按最近编辑排序
```

#### US-2.2 章节管理

```
作为一个长篇作者，
我想要将内容分章节管理，
以便更好地组织结构。

验收标准：
- [ ] 可以创建/重命名/删除章节
- [ ] 章节可以拖拽排序
- [ ] 每章有字数统计
- [ ] 支持章节间快速切换
```

#### US-2.3 导出分享

```
作为一个作者，
我想要导出我的作品，
以便在其他平台发布。

验收标准：
- [ ] 支持导出 Markdown / Word / PDF
- [ ] 支持选择导出全部或部分章节
- [ ] 支持生成分享链接（只读）
```

### 4.3 Level 3: 世界观管理

#### US-3.1 角色卡片

```
作为一个小说作者，
我想要管理故事中的角色，
以便在写作时保持一致性。

验收标准：
- [ ] 可以创建角色卡片（姓名、描述、性格、关系）
- [ ] 写作时可以快速引用角色
- [ ] AI在续写时会考虑角色设定
- [ ] 可以从已有文本提取角色
```

#### US-3.2 设定库

```
作为一个奇幻/科幻作者，
我想要记录世界观设定，
以便保持故事内在逻辑。

验收标准：
- [ ] 可以创建地点、物品、势力等设定
- [ ] 设定可以分类管理
- [ ] AI在写作时遵循设定约束
```

### 4.4 Level 4: AI团队协作

#### US-4.1 多视角审稿

```
作为一个希望提升质量的作者，
我想要让多个AI角色审阅我的文章，
以便获得不同视角的反馈。

验收标准：
- [ ] 点击"获取反馈"
- [ ] 展示3个AI角色的评价（编辑视角、读者视角、专业视角）
- [ ] 每个角色给出具体建议
- [ ] 可以一键应用建议
```

#### US-4.2 一致性检查

```
作为一个长篇作者，
我想要检查全文的一致性问题，
以便避免穿帮。

验收标准：
- [ ] 一键运行一致性检查
- [ ] 展示发现的问题（角色、时间线、设定矛盾）
- [ ] 每个问题可以定位到原文
- [ ] 提供修改建议
```

### 6.3 API设计

#### 6.3.1 核心API

```typescript
// 流式续写（Level 1核心功能）
POST /api/v1/ai-writing/stream/continue
Body: {
  content: string;           // 当前内容
  instruction?: string;      // 可选指令
  projectId?: string;        // 如果在项目中
  worldContext?: {           // 可选世界观上下文
    characters?: string[];
    settings?: string[];
  }
}
Response: SSE stream

// 文本操作（Level 1）
POST /api/v1/ai-writing/stream/transform
Body: {
  text: string;              // 选中的文本
  action: 'expand' | 'condense' | 'rewrite' | 'formalize' | 'casualize';
  customInstruction?: string;
  context?: string;          // 前后文
}
Response: SSE stream

// 获取建议（Level 1）
POST /api/v1/ai-writing/suggestions
Body: {
  content: string;           // 当前内容
  cursorPosition?: number;   // 光标位置
}
Response: {
  suggestions: Array<{
    title: string;
    description: string;
    preview?: string;
  }>
}

// 项目管理（Level 2）
POST /api/v1/ai-writing/projects
GET  /api/v1/ai-writing/projects
GET  /api/v1/ai-writing/projects/:id
PUT  /api/v1/ai-writing/projects/:id
DELETE /api/v1/ai-writing/projects/:id

// 章节管理（Level 2）
POST /api/v1/ai-writing/projects/:id/chapters
PUT  /api/v1/ai-writing/chapters/:id
```
