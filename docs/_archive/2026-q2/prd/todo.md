# 产品规划 TODO

这份文档的目标不是做方向罗列，而是把当前产品规划拆成可推进、可分工、可验收的任务清单。原则是：

- 每一项都要能落到代码、流程或产品交付
- 每一项都要有清晰边界，避免“越做越散”
- 每一项都要有验收标准，避免口号式推进

---

## P0：架构基建

### 1. Token 压缩机制

**目标**

建立统一的上下文压缩层，降低长链路任务的 token 消耗，并保证压缩后的任务仍能稳定生成高质量结果。

**要解决的问题**

- 多 Agent 长链路任务中，上下文越来越长，成本越来越高
- 同一份原始信息在多个阶段重复传递，存在大量冗余
- 目前压缩逻辑分散，不同流程的压缩行为不一致，不可观测

**推进重点**

1. 定义统一压缩入口
   - 明确哪些场景必须经过压缩层
   - 明确压缩前后数据结构
   - 保证所有长链路任务走同一套压缩 contract

2. 建立多级压缩策略
   - 原文保留
   - 摘要压缩
   - 结构化事实提取
   - 章节级 / 证据级 / 任务级不同粒度压缩

3. 建立压缩质量守护
   - 压缩后不得丢失关键结论、证据链接、约束条件
   - 对高风险字段设置强保留规则
   - 对压缩结果增加可回放和比对能力

4. 接入运行时观测
   - 记录压缩前 token、压缩后 token、压缩率
   - 记录压缩是否引发质量下降
   - 记录不同链路的压缩命中情况

**代码落点建议**

- `ai-engine`：压缩算法、摘要器、结构化抽取器
- `ai-harness`：workflow 中的统一压缩接入点
- `ai-app`：只保留压缩策略选择，不自行实现压缩逻辑

**验收标准**

- 长链路 mission 的 token 消耗明显下降
- 质量不因压缩显著退化
- 所有关键任务都能观测压缩前后差异
- 压缩逻辑不再散落在多个业务模块中

### 2. 分层架构重构（泛化下沉 / 业务上移）

**目标**

把通用能力沉淀到平台层，把产品专属逻辑收敛到 APP 层，形成可复用、可扩展、可裁剪的整体架构。

**分层边界**

- `ai-infra`
  - 数据库、缓存、队列、配置、鉴权、文件存储、第三方连接器、计费等基础设施

- `ai-engine`
  - 模型调用、工具系统、提示词、知识处理、RAG、内容生成、单 Agent 通用能力

- `ai-harness`
  - 多 Agent runtime、workflow 编排、状态机、review / critique / remediation、质量守门、协作协议、运行时观测

- `ai-app`
  - 产品语义、业务流程、页面状态、产品数据聚合、产品专属规则

**核心原则**

- 可脱离 Playground 语义独立成立的能力，优先下沉
- 带 mission、chapter、leader/researcher、admin、产品页面状态的逻辑，保留在 APP
- 先定义 contract，再迁移实现

**优先迁移对象**

1. Report artifact 生成、切片、校验、修复
2. 通用 workflow 状态机与 retry / rerun 机制
3. Reviewer / Critic / Remediation / Quality gate
4. Token / cost / usage / runtime observability
5. Team 协作骨架与任务调度协议

**推进阶段**

1. 边界梳理
   - 盘点通用能力与业务耦合点
   - 标记保留层与迁移层
   - 定义 contract

2. 能力下沉
   - 先迁低耦合高复用模块
   - 补统一测试与文档
   - 保证迁移前后行为一致

3. APP 收薄
   - 清理 APP 对底层内部实现的穿透引用
   - 让 APP 只保留组装与业务规则
   - 用第二个调用方验证复用能力

**验收标准**

- 新增一个 APP 时不需要复制 Playground 内核代码
- 通用能力至少被两个场景复用
- `ai-app` 不再实现本应属于 `ai-harness` / `ai-engine` 的底层逻辑
- Playground 删掉一批历史通用实现后，无功能回归

### 3. CLI 全链路支持

**目标**

提供端到端 CLI，使团队可以在不依赖前端 UI 的情况下完整驱动任务、观察执行、导出结果、做自动化验证。

**要解决的问题**

- 当前很多能力只能靠页面点击验证
- 本地复现、CI 验证、容器 smoke test 成本高
- 同一条任务在不同环境下缺少统一调试入口

**推进重点**

1. 定义 CLI 能力边界
   - 启动 mission
   - 查看中间事件流
   - 查看最终 artifact
   - 查看 token/cost/质量评分
   - 导出结果

2. 对接平台层能力
   - CLI 不直接写业务逻辑
   - 通过 `ai-harness / ai-engine / ai-app` 暴露的稳定接口驱动

3. 支持本地与容器环境
   - 本地开发环境
   - Docker 容器内运行
   - CI 环境自动执行

4. 建立标准脚本
   - smoke test
   - 回归测试
   - 基准任务跑批
   - 问题复现实验

**验收标准**

- 任意一条标准 mission 可通过 CLI 完整跑通
- CLI 能输出中间进度、最终结果和资源消耗
- 本地、容器、CI 使用同一套命令入口
- 问题复现不再依赖 UI 手工操作

## P1：能力复用

### 4. Team 模式订阅

**目标**

把多 Agent 协作模式产品化为可订阅能力包，使新 APP 能直接复用协作模板，而不是重复搭建。

**要解决的问题**

- 多 Agent 结构在多个场景中重复实现
- Team 配置与业务逻辑强耦合，难以迁移
- 新产品接入成本高

**推进重点**

1. 抽象 Team 能力包
   - 角色集合
   - 协作关系
   - 阶段顺序
   - 质量门槛
   - 失败恢复策略

2. 建立 Team 配置模型
   - 支持版本化
   - 支持参数化
   - 支持按场景裁剪

3. 支持 APP 装配
   - APP 通过选 Team 模式完成大部分编排接入
   - APP 只补业务特有字段与页面层配置

**验收标准**

- 至少有两类 APP 共用 Team 模式
- Team 调整不需要改动业务页面大量代码
- Team 配置可独立发布、复用、升级

---

### 5. Workflow 订阅与可视化编排

**目标**

提供预置 Workflow 订阅能力，同时支持用户通过可视化或对话式方式自行编排，并生成可执行 Agent 工作流。

**要解决的问题**

- Workflow 目前偏工程实现，产品化程度不够
- 新流程创建成本高
- 非工程角色无法自主配置流程

**推进重点**

1. 平台预置 Workflow
   - 提供标准流程模板
   - 支持按行业、任务类型、复杂度区分

2. UI 可视化编排
   - 节点配置
   - 依赖关系
   - 角色选择
   - 输入输出定义

3. 对话式编排
   - 用户通过自然语言描述流程目标
   - 系统自动生成初版 workflow
   - 用户再调整细节

4. 编排结果落地
   - 生成可执行 Agent 配置
   - 接入平台 runtime
   - 可直接复用于 APP 或 Team

**验收标准**

- 至少一条平台预置 workflow 可被多个场景订阅
- 用户可通过 UI 或对话式方式生成可执行 workflow
- workflow 配置变更不需要手工改大量代码

### 6. 本地模型支撑

**目标**

保证系统在本地模型环境下也能跑通完整链路，并稳定产出可接受质量的报告。

**要解决的问题**

- 当前很多能力默认依赖远端强模型
- 本地模型在长上下文、工具调用、结构化输出方面稳定性差
- 缺少专门针对本地模型的质量守护和评测体系

**推进重点**

1. 梳理本地模型兼容性
   - 哪些流程必须调整 prompt
   - 哪些流程需要更强约束
   - 哪些工具调用需要降级

2. 建立本地模型专用策略
   - 更保守的 workflow
   - 更短的上下文
   - 更严格的 schema / retry / fallback

3. 建立本地模型评测集
   - 完整性
   - 正确性
   - 稳定性
   - 成本与延迟

**验收标准**

- 本地模型可跑通标准任务链路
- 报告完整性与基本质量达标
- 本地模型与远端模型的差异可被量化观测

---

### 7. Skill 导入与报告生成

**目标**

支持导入外部 Skill，并基于 Skill 结果输出不同类型的报告成品。

**要解决的问题**

- Skill 与业务产出之间链路不闭环
- 导入 Skill 后难以直接形成高价值成果
- 报告形态支持不统一

**推进重点**

1. Skill 导入标准化
   - Skill 元数据
   - 输入输出协议
   - 权限与依赖声明

2. Skill 与 workflow / Team 打通
   - Skill 可被节点或角色直接调用
   - Skill 输出可进入后续报告生成链路

3. 多形态报告生成
   - 文档
   - PPT
   - 可视化看板
   - 可复用数据摘要

**验收标准**

- 至少一种外部 Skill 可标准化导入并接入主流程
- Skill 输出可直接进入报告生成
- 至少支持两种稳定报告形态

## P1：质量与自动化

### 8. 100% 自动化

**目标**

实现端到端自动化运行，使任务从输入、执行、评估到输出全过程不依赖人工中间操作。

**要解决的问题**

- 当前链路中仍有较多手工触发、手工确认、手工排障环节
- 自动化覆盖率不足，导致规模化运行困难

**推进重点**

1. 自动化运行链路梳理
   - 输入准备
   - 调度执行
   - 中间失败恢复
   - 输出落库与导出

2. 异常自动处理
   - 重试
   - fallback
   - 降级
   - 终止与告警

3. 自动化验证
   - 任务结束后自动跑完整性检查
   - 质量门控不过自动阻断
   - 关键指标自动记录

**验收标准**

- 标准任务全程无需人工干预
- 常见失败能自动恢复或明确终止
- 自动化链路可在本地、容器、CI 中稳定复用

## P2：闭环与扩展

### 9. 内容发布与反馈闭环

**目标**

把内容生成、发布、反馈回收和优化串成闭环，形成真正能持续学习和迭代的系统。

**要解决的问题**

- 现在大多只做到“生成内容”，没有系统化回收真实反馈
- 缺少反馈驱动的内容优化机制

**推进重点**

1. 接入发布平台
   - 公众号
   - 小红书
   - 后续其他渠道

2. 回收反馈数据
   - 阅读
   - 点赞
   - 收藏
   - 评论
   - 转化

3. 建立反馈到优化链路
   - 反馈进入分析层
   - 形成新的 prompt / workflow / content strategy 调整
   - 支持自动与人工共同闭环

**验收标准**

- 至少一个发布渠道形成自动发布能力
- 能自动回收核心反馈指标
- 反馈数据能进入后续优化流程

---

### 10. 多模态输入

**目标**

支持更自然的输入方式，扩大产品适用场景。

**推进重点**

1. 语音输入
   - 语音转文本
   - 上下文理解
   - 与现有任务输入协议兼容

2. 视频输入评估
   - 明确适用场景
   - 明确收益与成本
   - 决定是否进入正式开发

**验收标准**

- 语音输入可进入标准任务链路
- 视频输入有明确的 ROI 评估结论，而不是模糊挂起

## 当前建议的落地方式

如果要真正推进，下一步不应该继续扩写概念，而应该从这份 TODO 再拆出两层：

1. `Roadmap`
   - 按季度或月份排优先级和里程碑

2. `Execution Plan`
   - 每项拆成具体任务、负责人、预估工期、代码落点、验收方式

也就是说，这份文档现在已经可以作为“规划母稿”，但还不是研发排期表。下一步应从这里继续拆成执行级计划。
