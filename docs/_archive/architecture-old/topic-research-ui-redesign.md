## 2. 核心架构理念：Leader 自主决策 + 团队自组织

> **核心思想**：不预设任何 Workflow，由 Leader（最强推理模型）自主理解任务、定义工作流、分配 Agent。用户通过 UI 提示补充 Leader 盲区，但不直接干预执行流程。

### 2.1 架构设计原则

```
┌─────────────────────────────────────────────────────────────────────┐
│                       【用户】                                       │
│                          │                                          │
│                    提供研究目标                                      │
│                    补充提示/约束                                     │
│                          │                                          │
│                          ▼                                          │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    👑 Leader                                 │   │
│  │              (最强推理模型: Claude Opus / o1)                │   │
│  │                                                              │   │
│  │  职责：                                                      │   │
│  │  1. 任务理解 - 深度分析研究目标和范围                        │   │
│  │  2. 工作流定义 - 自主决定研究维度和阶段划分                  │   │
│  │  3. Agent 分配 - 指定具体 Agent 负责特定章节/维度            │   │
│  │  4. 质量把关 - 审核所有输出，决定是否需要修订                │   │
│  │  5. 最终整合 - 汇总各 Agent 成果，生成完整报告               │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                          │                                          │
│           ┌──────────────┼──────────────┐                          │
│           ▼              ▼              ▼                          │
│      ┌────────┐    ┌────────┐    ┌────────┐                        │
│      │ 🔍 研究│    │ 🔍 研究│    │ 🔍 研究│  ← Leader 动态创建     │
│      │ Agent A│    │ Agent B│    │ Agent C│    按需分配任务        │
│      └────────┘    └────────┘    └────────┘                        │
│           │              │              │                          │
│           └──────────────┼──────────────┘                          │
│                          ▼                                          │
│                    ┌──────────┐                                     │
│                    │ ✅ 质量  │  ← Leader 决定是否需要              │
│                    │   审核   │     以及由谁执行                    │
│                    └──────────┘                                     │
│                          │                                          │
│                          ▼                                          │
│                    ┌──────────┐                                     │
│                    │ 📊 报告  │  ← Leader 可能自己写                │
│                    │   撰写   │     或指定 Agent 写                 │
│                    └──────────┘                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 与传统预设 Workflow 的区别

| 维度           | 传统预设 Workflow               | Leader 自组织       |
| -------------- | ------------------------------- | ------------------- |
| **流程定义**   | 开发者硬编码                    | Leader 动态决定     |
| **Agent 数量** | 固定（如 5 个角色）             | 按需创建，弹性调整  |
| **任务分配**   | 预定义规则                      | Leader 实时判断     |
| **执行顺序**   | 固定流程（规划→采集→审核→撰写） | Leader 根据需要调整 |
| **质量控制**   | 固定审核点                      | Leader 全程把关     |
| **用户干预**   | 通过配置参数                    | 通过提示补充 Leader |

### 2.3 Leader 的核心能力要求

```
┌──────────────────────────────────────────────────────────────┐
│  Leader 模型配置                                             │
├──────────────────────────────────────────────────────────────┤
│  默认模型: Claude Opus 4 / OpenAI o1                        │
│  modelType: AIModelType.REASONING                            │
│  TaskProfile:                                                │
│    - creativity: "medium"     (允许创造性规划)               │
│    - outputLength: "extended" (支持复杂推理输出)             │
│                                                              │
│  关键能力:                                                   │
│  ✓ 复杂任务分解                                              │
│  ✓ 多步骤推理规划                                            │
│  ✓ 动态资源调度                                              │
│  ✓ 质量评估与决策                                            │
└──────────────────────────────────────────────────────────────┘
```

### 2.4 用户角色：提示补充者（非流程控制者）

**用户可以做的：**

- 📝 提供研究目标和背景
- 💡 补充 Leader 可能忽略的角度/约束
- 🔄 请求 Leader 调整方向（如"更关注技术细节"）
- ❓ 询问当前进度和 Leader 的决策理由

**用户不需要做的：**

- ❌ 手动配置 Agent 数量和角色
- ❌ 定义工作流步骤
- ❌ 指定哪个 Agent 做哪个任务
- ❌ 管理 Agent 之间的协调

## 3. UI 重设计目标

### 3.1 核心原则

1. **Leader 决策可视化** - 用户能看到 Leader 的任务分解和分配决策
2. **团队自组织透明** - 实时展示 Agent 创建、分配、执行状态
3. **用户提示入口突出** - @Leader 是唯一的交互入口
4. **不暴露流程控制** - UI 不提供手动干预 Agent 执行的功能

### 3.2 用户旅程重塑

```
创建专题 → 描述目标 → Leader 规划 → 观察执行 → 补充提示 → 查看报告
    ↓          ↓           ↓            ↓           ↓          ↓
 对话框    @Leader    自动展示      团队互动    @Leader    洞察报告
           输入目标   思考架构       Tab        补充提示
```

// Leader 动态创建研究团队
interface DynamicResearchTeam {
  leader: LeaderAgent;
  researchers: ResearcherAgent[];  // 按维度动态创建
  analyst?: AnalystAgent;          // 可选
  writer?: WriterAgent;            // 可选
}

// Leader 决定需要哪些 Agent
async createTeamForTopic(topic: Topic, leaderPlan: LeaderPlan) {
  const team: DynamicResearchTeam = {
    leader: await this.createLeaderAgent(topic),
    researchers: [],
  };

  // 根据 Leader 规划动态创建研究员
  for (const dimension of leaderPlan.dimensions) {
    const researcher = await this.createResearcherAgent(dimension);
    team.researchers.push(researcher);
  }

  // Leader 决定是否需要专门的分析师和撰稿人
  if (leaderPlan.needsAnalyst) {
    team.analyst = await this.createAnalystAgent();
  }
  if (leaderPlan.needsWriter) {
    team.writer = await this.createWriterAgent();
  }

  return team;
}

### Leader 模型配置（后端）

> **架构说明**：AI App 定义诉求（modelType），AI Engine 从用户配置的数据库中选择具体模型。

```typescript
// ==================== Leader 模型获取方式 ====================

// 方式 1: 使用现有 AIModelType.CHAT，需要用户配置推理模型为默认
// 用户在 AIModel 表中设置：modelType=CHAT, isDefault=true, isReasoning=true
await this.aiChatService.chat({
  messages,
  modelType: AIModelType.CHAT,  // 使用用户配置的默认 CHAT 模型
  taskProfile: {
    creativity: "medium",       // 允许创造性规划
    outputLength: "extended",   // 支持复杂推理输出（8000+ tokens）
  },
});

// 方式 2: 新增获取推理模型的方法（推荐，需要 AI Engine 新增）
// ResearchLeaderService 中调用
async getLeaderModel(): Promise<string> {
  // 优先获取 isReasoning=true 的模型
  const reasoningModel = await this.aiChatService.getReasoningModel();
  if (reasoningModel) return reasoningModel.modelId;

  // 回退到默认 CHAT 模型
  const defaultModel = await this.aiChatService.getDefaultModelByType(AIModelType.CHAT);
  return defaultModel?.modelId || 'claude-opus';
}

// ==================== 需要 AI Engine 新增的方法 ====================

// 在 AiChatService 中新增：
async getReasoningModel(): Promise<AIModelConfig | null> {
  // 查找 isReasoning=true 的启用模型
  const model = await this.prisma.aIModel.findFirst({
    where: {
      modelType: AIModelType.CHAT,
      isEnabled: true,
      isReasoning: true,
    },
    orderBy: [
      { isDefault: 'desc' },  // 优先默认推理模型
      { createdAt: 'desc' },  // 其次最新添加的
    ],
  });

  if (model) {
    return this.toModelConfig(model);
  }
  return null;
}

// ==================== TaskProfile 说明 ====================
// creativity: "medium" → temperature: 0.7（允许创造性规划）
// outputLength: "extended" → maxTokens: 16000（推理模型自动调整为 8000+）
// 推理模型（o1/o3）会自动设置 reasoning_effort 参数
```

**用户配置要求**（数据库 AIModel 表）：

| 字段          | 值                      | 说明             |
| ------------- | ----------------------- | ---------------- |
| `modelType`   | `CHAT`                  | 文本聊天类型     |
| `modelId`     | `claude-opus-4` 或 `o1` | 具体模型 ID      |
| `isEnabled`   | `true`                  | 启用             |
| `isReasoning` | `true`                  | 标记为推理模型   |
| `isDefault`   | `true`（可选）          | 设为默认推理模型 |

**推荐的推理模型优先级**：

1. Claude Opus 4（claude-opus-4-5-20251101）
2. OpenAI o1 / o3
3. Claude Sonnet（回退）
