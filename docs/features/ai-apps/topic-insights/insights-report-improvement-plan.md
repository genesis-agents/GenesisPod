## 二、优化方案

### Phase 1: 修复致命缺陷 (P0)

#### 1.1 修复执行摘要生成

**问题根因**: 合成阶段输入内容过长，触发 LLM 安全护栏的 `input-complexity-check`。

**解决方案**:

```
当前流程:
  全部维度内容 → 一次性送入 LLM → 触发 guard → 报错

优化流程:
  全部维度内容 → 先提取每个维度的摘要 (已有 summary 字段)
                → 仅用摘要+关键数据生成执行摘要
                → 避免触发输入复杂度检查
```

**涉及文件**:

- `backend/src/modules/ai-app/topic-insights/services/report/report-synthesis.service.ts`
- `backend/src/modules/ai-app/topic-insights/prompts/report-synthesis.prompt.ts`

**具体改动**:

1. 合成执行摘要时，不传入 `detailedContent`，仅传入 `summary` + `keyFindings` + `trends`
2. 为执行摘要生成增加 fallback: 若结构化生成失败，降级为基于 `summary` 字段的简单拼接
3. 执行摘要应输出为 `ExecutiveSummaryObject` 结构化格式，而非纯文本

**执行摘要目标格式**:

```markdown
## 执行摘要

### 核心结论

1. [一句话结论，不超过30字]
2. [一句话结论，不超过30字]
3. [一句话结论，不超过30字]

### 关键数据

| 指标                 | 数值    | 来源                |
| -------------------- | ------- | ------------------- |
| 美国AI市场规模(2025) | $173.5B | Precedence Research |
| AI创业VC占比(2025)   | 65.6%   | CB Insights         |
| ...                  | ...     | ...                 |

### 风险提示

- [风险1]
- [风险2]

### 行动建议

- [建议1]
- [建议2]
```

#### 1.2 报告篇幅控制

**核心策略**: 引入**分层报告体系**，而非压缩单一报告。

```
┌──────────────────────────────────────────────┐
│  精华版 (Executive Brief)                     │
│  5-8 页 | 核心结论 + 关键图表 + 行动建议       │
│  目标读者: C-Level / 决策者                    │
├──────────────────────────────────────────────┤
│  标准版 (Standard Report)                     │
│  30-50 页 | 各章摘要 + 分析 + 图表 + 建议      │
│  目标读者: VP/Director / 项目负责人            │
├──────────────────────────────────────────────┤
│  完整版 (Full Research)                       │
│  当前版本 | 深度细节 + 全部引注 + 附录          │
│  目标读者: 研究分析师 / 领域专家               │
└──────────────────────────────────────────────┘
```

**实现路径**:

A. **短期 (v1.1)**: 在现有报告顶部增加折叠式精华摘要

- 前端增加 "Quick View" 模式，默认展示精华内容
- 每个章节增加折叠/展开控制

B. **中期 (v2.0)**: 后端生成分层版本

- 报告合成时同步生成精华版和标准版
- 精华版: 执行摘要 + 每章 Key Takeaways + 跨维度分析 + 建议
- 标准版: 精华版 + 各章节核心分析段落 + 全部图表

**涉及文件**:

- `backend/src/modules/ai-app/topic-insights/types/report.types.ts` - 新增 `ReportTier` 类型
- `backend/src/modules/ai-app/topic-insights/services/report/report-generator.service.ts` - 分层生成逻辑
- 前端报告渲染组件 - 增加版本切换 UI

**新增类型定义** (`report.types.ts`):

```typescript
/** 报告层级 */
export type ReportTier = "executive" | "standard" | "full";

/** 精华版报告 */
export interface ExecutiveBriefReport {
  executiveSummary: ExecutiveSummaryObject;
  chapterHighlights: Array<{
    sectionNumber: string;
    title: string;
    keyTakeaways: string[]; // 3-5条
    keyDataPoint: { metric: string; value: string; source: string };
    outlook: string; // 一句话展望

  crossDimensionInsight: string; // 300字以内
  topRisks: Array<{ risk: string; probability: string; impact: string }>;
  actionItems: Array<{
    target: "enterprise" | "investor" | "policymaker";
    timeframe: "short" | "mid" | "long";
    action: string;
  }>;
}
```

### Phase 2: 提升内容质量 (P1)

#### 2.1 消除文字墙 - 引入结构化内容块

**问题**: 所有内容以密集长段落呈现，缺乏视觉停顿。

**解决方案**: 在维度研究阶段的 prompt 中引入**内容块 (Content Block)** 概念。

**内容块类型**:

| 块类型             | 用途          | 渲染样式             |
| ------------------ | ------------- | -------------------- |
| `key-finding`      | 核心发现高亮  | 蓝色左边框卡片       |
| `data-highlight`   | 关键数据突出  | 数据卡片 (数值+来源) |
| `comparison-table` | 对比分析      | 结构化表格           |
| `timeline`         | 时间线事件    | 时间轴可视化         |
| `callout-warning`  | 风险/注意事项 | 黄色警告框           |
| `callout-insight`  | 独特洞察      | 紫色洞察框           |
| `quote`            | 关键引述      | 引用样式             |

**Prompt 优化示例** (维度研究阶段):

```markdown
## 内容格式要求

每个小节必须包含以下结构元素（不是全部，按需使用）：

1. **开头**: 一句话核心论点（Bold-Bullet 格式）
2. **正文**: 2-3 段分析，每段不超过 150 字
3. **数据支撑**: 至少一个 `:::data-highlight` 块
4. **小结**: 一句话 "So What" 结论

### 内容块语法

:::key-finding
美国AI政策呈现联邦-州二元分裂格局，2026年将出现激烈的法律冲突。
:::

:::data-highlight
| 指标 | 数值 | 来源 |
|------|------|------|
| AI创业VC占比 | 65.6% | CB Insights 2025 |
:::

:::callout-warning
特朗普政府的新行政命令可能导致联邦与州级法律的全面冲突。
:::
```

**涉及文件**:

- 维度研究 prompt 文件（各维度 Agent 的系统提示词）
- 前端 Markdown 渲染组件 - 新增自定义块解析和渲染
- `report.types.ts` - 无需改动（内容块嵌入 Markdown 内容中）

#### 2.2 引注规范化

**当前问题**: `[174] [19] [26] [18] [28]` 大量内联引注堆砌在段落中。

**解决方案**: 引注压缩 + 悬浮预览

**规则**:

1. 同一句话最多保留 **2 个**引注
2. 超过 2 个引注时，合并为 `[19 等5项]`，hover 展示完整列表
3. 精华版和标准版完全移除内联引注，改为章节末尾引用列表

**实现方式**:

A. **Prompt 端**: 在维度研究 prompt 中增加引注约束:

```markdown
## 引注规范

- 每个论述点最多引用2个来源，选择最权威的
- 如有多个来源支持同一观点，使用 [来源A; 来源B 等N项] 格式
- 不要在一句话中堆砌超过3个引注编号
```

B. **渲染端**: 前端对引注进行智能折叠:

- 连续 3+ 个引注自动折叠
- 点击展开完整引注列表
- hover 显示来源摘要

#### 2.3 图表体系升级

**当前问题**: 仅有简单柱状图 (2-4 个数据点)，缺乏多样性和深度。

**图表类型扩展矩阵**:

| 分析场景 | 推荐图表类型  | 当前支持  | 需新增       |
| -------- | ------------- | --------- | ------------ |
| 趋势变化 | 折线图/面积图 | line/area | -            |
| 规模对比 | 柱状图/条形图 | bar       | -            |
| 占比分布 | 饼图/环形图   | pie       | 环形图变体   |
| 多维对比 | 雷达图        | radar     | -            |
| 关联关系 | 散点图        | -         | **scatter**  |
| 流向关系 | 桑基图        | -         | **sankey**   |
| 竞争格局 | 四象限图      | -         | **quadrant** |
| 风险评估 | 热力图/矩阵   | -         | **heatmap**  |
| 时间线   | 甘特图/里程碑 | -         | **timeline** |
| 层级关系 | 树图          | -         | **treemap**  |
| 地理分布 | 地图          | -         | **geo-map**  |

**优先实现 (Phase 2)**:

1. **四象限图** (quadrant) - 用于竞争格局、风险评估
2. **热力图** (heatmap) - 用于风险矩阵、能力对比
3. **桑基图** (sankey) - 用于跨维度关联、资金流向
4. **时间线** (timeline) - 用于政策演进、技术里程碑

**涉及文件**:

- `report.types.ts` - 扩展 `ChartType` 类型
- 前端图表渲染组件 - 新增图表类型 (基于 Recharts 或 D3)
- 维度研究 prompt - 引导生成更多样的图表数据

**类型扩展**:

```typescript
export type ChartType =
  | "line"
  | "bar"
  | "area"
  | "pie"
  | "radar"
  | "composed" // 现有
  | "scatter"
  | "heatmap"
  | "sankey"
```

### Phase 3: 提升专业度 (P2)

#### 3.1 统一章节结构框架

每个主题章节应遵循统一的结构模板 (**Key Insights Framework**):

```markdown
## N. [章节标题]

:::key-takeaways

### 核心发现

1. [发现1 - 一句话]
2. [发现2 - 一句话]
3. [发现3 - 一句话]
   :::

### N.1 [子主题1]

[分析正文 - Bold-Bullet 格式]
[内嵌图表]

### N.2 [子主题2]

[分析正文]
[内嵌图表]

...

### N.x 本章总结与展望

:::chapter-summary
| 维度 | 评估 |
|------|------|
| 当前成熟度 | [评分/描述] |
| 2026年趋势 | [上升/平稳/下降] |
| 关键风险 | [一句话] |
| 关键机会 | [一句话] |
| 建议行动 | [一句话] |
:::
```

**实现方式**: 修改维度研究 Agent 的输出模板，在 prompt 中强制要求遵循此框架。

#### 3.2 跨维度关联分析升级

**当前问题**: 仅罗列各维度摘要，标注字数 (如 "72字")，缺乏真正的关联分析。

**目标格式**:

```markdown
## 跨维度关联分析

### 因果链路图

[桑基图/网络图: 展示维度间的因果关系]

### 核心联动机制

#### 联动链 1: 政策 → 资本 → 技术 → 市场

- **触发点**: 特朗普政府新行政命令 (政策法规)
- **传导路径**: 联邦优先权确立 → 合规成本降低 → VC加速投入 → 技术商业化提速
- **时间窗口**: 2026 Q1-Q3
- **确定性**: 中等（取决于国会立法进度）
- **受益方**: 大型科技公司 > AI创业公司

#### 联动链 2: 算力 → 人才 → 应用

...

### 维度关联矩阵

[热力图: 10x10 维度间关联强度]
```

**Prompt 优化**: 跨维度分析 prompt 需要:

1. 明确要求输出因果链路（不是摘要罗列）
2. 每条链路包含: 触发点、传导路径、时间窗口、确定性评级
3. 生成关联矩阵数据（维度两两关联强度评分）

#### 3.3 风险评估结构化

**当前问题**: 风险评估为纯叙述文字，缺乏量化框架。

**目标格式**:

```markdown
## 风险评估

### 风险矩阵

[热力图: 概率 x 影响 矩阵，标注各风险点位]

### 关键风险清单

| #   | 风险            | 概率 | 影响 | 时间窗口  | 预警指标       | 缓解策略       |
| --- | --------------- | ---- | ---- | --------- | -------------- | -------------- |
| R1  | 联邦-州法律冲突 | 高   | 高   | 6个月内   | 州级AI法案数量 | 分区域合规策略 |
| R2  | AI投资泡沫      | 中   | 高   | 12-18个月 | 资产溢价率     | 分散投资配置   |
| ... | ...             | ...  | ...  | ...       | ...            | ...            |

### 风险演进路线图

[时间线图: 各风险在未来24个月的预期演进]
```

**已有基础**: `riskAssessment` 类型已定义结构化字段 (`riskMatrix`, `probability`, `impact` 等)，但前端渲染时降级为纯文本 `fullText`。

**修复方向**:

1. 确保 LLM 输出正确填充结构化字段 (不仅是 `fullText`)
2. 前端渲染时优先使用结构化数据生成热力图和表格
3. `fullText` 仅作为 fallback

#### 4.3 数据置信度标注

为关键数据点增加置信度标注:

```markdown
美国AI市场规模将在2035年达到 $976B ±15% [来源: Precedence Research] [置信度: 中]
```

置信度等级:

- **高**: 多个权威来源交叉验证
- **中**: 单一权威来源或多个一般来源
- **低**: 单一来源或预测性数据
- **推断**: 基于报告分析推断，无直接来源

## 五、验收标准

### Phase 1 完成标准

- [ ] 执行摘要正常生成，包含核心结论/关键数据/风险提示/行动建议
- [ ] 精华版报告可在 3 分钟内阅读完毕
- [ ] 报告支持章节折叠/展开

### Phase 2 完成标准

- [ ] 每个章节至少包含 1 个 Key Finding 卡片和 1 个数据高亮块
- [ ] 单段引注不超过 3 个，超出自动折叠
- [ ] 图表类型覆盖 4+ 种 (柱状/折线/四象限/热力图)

### Phase 3 完成标准

- [ ] 所有章节遵循 Key Insights Framework 模板
- [ ] 跨维度分析包含至少 2 条因果链路 + 关联矩阵图
- [ ] 风险评估包含概率-影响矩阵 (非纯文本)
- [ ] 战略建议按角色+时间框架表格化

### Phase 4 完成标准

- [ ] 报告包含方法论章节
- [ ] 参考文献按类型分组
- [ ] 关键数据标注置信度

### 整体验收

- [ ] 精华版报告篇幅 5-8 页
- [ ] 标准版报告篇幅 30-50 页
- [ ] 图表密度: 每 3-5 页至少 1 个
- [ ] 执行摘要可独立支持决策

## 六、风险与依赖

| 风险                           | 概率 | 影响 | 缓解措施                                    |
| ------------------------------ | ---- | ---- | ------------------------------------------- |
| LLM 输出不遵循结构化模板       | 中   | 高   | JSON Schema 强约束 + 输出验证 + retry       |
| 新图表类型前端实现复杂度高     | 中   | 中   | 优先用 Recharts 内置类型，复杂图表降级      |
| 内容块语法与现有 Markdown 冲突 | 低   | 中   | 使用 `:::` 围栏语法，不与标准 Markdown 冲突 |
| 分层报告增加生成时间           | 高   | 低   | 精华版从已有数据提取，不额外调用 LLM        |
