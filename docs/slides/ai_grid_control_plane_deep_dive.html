01 — 总览定位
控制平面：从「GPU 集群」到「统一推理平台」的关键跳跃

AI GRID 的硬件层（RTX PRO 6000 Blackwell GPU、BlueField DPU、Spectrum-X 以太网）和网络层解决的是「在哪里放置算力」的问题。但仅有硬件不够——没有控制平面，AI GRID 就只是一堆通过高速网络连接的孤立 GPU 集群。控制平面才是让这些分散资源表现为单一智能系统的核心。

NVIDIA 官方技术文档
NVIDIA AI Grid 参考设计白皮书明确指出：控制平面引入了确定性的、SLA 感知的 AI 负载路由层——它不是简单地将请求发送到最近或最空闲的节点，而是将模型端点视为分布式系统架构来处理，实时评估延迟、成本、模型能力、健康信号和容量来精确决定每个请求的执行位置。同时，它通过 KV-Cache 感知的模型路由来优化 LLM prompt 的模型放置与复用，将兼容请求导向缓存命中率最高的实例。
docs.nvidia.com/ai-grid/whitepapers/ai-grid-reference-design/control_plane_design.html
第三方分析 · Nscale 技术解析
Nscale 的分析文章进一步阐述：没有控制平面层，AI GRID 将只是一组由快速传输连接的 GPU 集群的集合。有了它，网格才能作为一个单一协调的智能系统运作。该控制平面需持续观察、优化和治理整个分布式环境。
nscale.com/blog/what-is-the-ai-grid
核心判断：控制平面是 AI GRID 与传统分布式计算（如 CDN、边缘计算）的根本差异所在。传统架构的流量调度是基于地理位置和负载的简单分发；AI GRID 控制平面是基于 AI 负载语义、KV-Cache 状态、SLA 约束和 Token 经济学的多维智能决策。
02 — 三层组件架构
AI Grid Orchestrator + 全局负载均衡器 + LLM 路由器

根据 NVIDIA 官方控制平面白皮书，控制平面被定义为一个「高性能神经系统」，由三个核心组件协同工作：

核心组件 ①
AI Grid Orchestrator（编排器）
维护全局状态——哪些集群健康、哪里有可用容量。管理 Pod、集群和模型部署的完整生命周期。若某节点心跳丢失，编排器自动检测并在其他位置重新配置模型。通常构建于 Kubernetes 或专用 AI 调度框架之上，包含决策与策略引擎。
↕
核心组件 ②
Global Load Balancer（全局负载均衡器）
处理入站数据流（Ingress），决定请求应被发送到哪里处理。分发请求到多个活跃实例，维护基于服务商设定标准的排名列表。查看硬件级健康数据（连接数、CPU/GPU 利用率），可直接或通过 AI 网关连接到编排器获取资源和策略更新。
核心组件 ③
LLM Router（大模型路由器）
专注智能请求路由。分析传入查询的上下文、复杂度和元数据，将每个请求导向最合适的模型。与标准负载均衡器不同，LLM 路由器具备上下文感知能力——分析传入 prompt 以决定最高效的路径。还可实现语义缓存。
↕
底层执行引擎
NVIDIA Dynamo
简化和自动化分布式推理的复杂性——将推理的各个阶段（prefill/decode）分解到不同 GPU 上，智能路由请求以避免冗余计算，并通过数据缓存将 GPU 内存扩展到低成本存储层。
NVIDIA 官方技术文档
上述三层架构直接来自 NVIDIA AI Grid 参考设计白皮书的控制平面设计章节。白皮书明确指出：与传统负载均衡器不同，AI Grid 控制平面基于五个维度路由流量——Prompt 感知、资源与利用率感知、多租户、数据治理、安全性。
docs.nvidia.com/ai-grid/whitepapers/ai-grid-reference-design/control_plane_design.html

03 — KPI 感知路由机制
不是「最近优先」，而是「SLA 最优」

传统边缘计算和 CDN 的路由逻辑通常很简单：将请求发送到地理距离最近或负载最低的节点。AI GRID 控制平面完全颠覆了这一范式。

NVIDIA 技术博客 · GTC 2026
NVIDIA 技术博客详细描述了控制平面的两大核心路由能力：(1) KPI 感知路由——基于延迟要求、主权约束和成本来放置负载；(2) 资源感知放置——持续考虑节点健康、利用率和配额，在用户察觉尾延迟飙升之前避开过载或退化的站点，同时将兼容流量导向 KV-cache 命中率高的节点，以降低 Token 延迟和每请求 GPU 消耗。
developer.nvidia.com/blog/building-the-ai-grid-with-nvidia-orchestrating-intelligence-everywhere/

路由决策的多维 KPI 矩阵

控制平面在每个推理请求到达时，实时评估以下多个维度并做出全局最优决策：

KPI 维度 评估内容 优化目标
端到端延迟 RTT + 排队延迟 + 计算延迟（tokenization + prefill + decode） 确保语音交互 <500ms、视觉 AI <1s 的硬 SLO
Token 成本 GPU 利用率 × 带宽开销 × 出口成本 在满足 SLA 的前提下最小化每 Token 成本
数据主权 租户的数据驻留规则 × 司法管辖区要求 数据不出境，模型和日志保留在指定管辖区内
模型能力 目标节点上可用模型版本 × 参数量级 复杂任务路由到大模型节点，简单任务路由到轻量 SLM
KV-Cache 命中率 各节点 KV-Cache 状态 × 请求与缓存的重叠度 最大化缓存复用，减少重复计算
节点健康 GPU 利用率、内存压力、存储性能、网络延迟、电力、热量 在问题影响负载之前主动规避
与传统负载均衡的本质区别：传统 GSLB（全局服务器负载均衡）只看硬件级指标——连接数和 CPU/GPU 利用率。AI GRID 控制平面增加了 AI 负载语义层（prompt 复杂度分析 + KV-Cache 状态追踪 + 数据主权约束），这是分布式 AI 推理特有的需求。
04 — KV-CACHE 感知路由
控制平面中最精密的智能：避免重复计算的全局缓存调度

KV-Cache 感知路由是 AI GRID 控制平面中技术含量最高、对推理效率影响最大的单一机制。要理解它的价值，需要先理解 KV-Cache 在推理中的角色。

什么是 KV-Cache？为什么它如此重要？

在 LLM 响应用户 prompt 之前，模型必须构建对输入请求的上下文理解——这个过程会产生 KV-Cache（Key-Value 缓存），它存储了模型在推理过程中各层的激活值（key 和 value）。这个过程计算密集，且随输入大小呈二次方增长。复用 KV-Cache 可以避免从头重新计算，显著降低推理时间和计算资源消耗。

NVIDIA Dynamo 技术博客
NVIDIA Dynamo Smart Router 追踪大规模 GPU 集群中跨多节点和解耦部署的 KV-Cache 状态，并高效路由传入请求以最小化昂贵的重新计算。它将传入请求进行哈希处理并存储在基数树（Radix Tree）中，从而在大规模分布式推理环境中追踪 KV 位置。
developer.nvidia.com/blog/introducing-nvidia-dynamo-a-low-latency-distributed-inference-framework-for-scaling-reasoning-ai-models/

KV-Cache 路由的精确技术机制

1
请求 Token 化与分块
传入的 prompt 被转换为 token，然后划分为固定大小的块（如每块 16 或 64 个 token）。
2
块哈希与全局索引
每个 token 块被哈希为唯一标识符。KVPublisher 在每个 worker 上发出 KV-Cache 事件（创建/移除），KVIndexer 全局追踪这些事件，使用前缀树（prefix tree）维护精确的重叠信息。
3
重叠分数计算
当新请求到达时，Smart Router 计算传入请求与分布式集群中所有 GPU 内存上活跃的 KV-Cache 块之间的重叠分数（overlap score）。路由决策基于成本函数：
logit = kv_overlap_score_weight × potential_prefill_blocks + potential_active_blocks
4
多因子智能路由
综合考虑缓存命中率、负载均衡和 GPU 容量，将请求智能路由到最合适的 worker。与轮询或基于负载的路由不同，这种方法优化了整个系统性能。权重参数 kv_overlap_score_weight 可调：较高值优先减少 prefill 成本（改善 TTFT），较低值优先均匀分布 decode 负载（改善 ITL）。
NVIDIA Dynamo 路由器文档
Dynamo KV Router 通过评估不同 worker 上的计算成本来智能路由请求。它同时考虑 decode 成本（来自活跃块）和 prefill 成本（来自新计算块），利用 KV-Cache 重叠来最小化冗余计算。路由器使用来自 worker 的 KV 事件来默认维护缓存块的精确全局视图。
docs.nvidia.com/dynamo/latest/user-guides/kv-cache-aware-routing
NVIDIA Dynamo GitHub 文档
在 Dynamo 架构中，KV-Cache 路由使用「直接路由」（direct routing）配合特殊的 worker 选择算法。SGLang 贡献的 RadixAttention 引入了前缀树结构，实现了 KV-Cache 块的高效匹配、插入和驱逐。Dynamo 在此基础上引入了 KVPublisher（在每个 worker 上发出 KV-Cache 事件）和 KVIndexer（全局追踪这些事件）。
github.com/ai-dynamo/dynamo · docs/kv_cache_routing.md
05 — NVIDIA DYNAMO 分布式推理引擎
控制平面的执行底座：五大核心组件

Dynamo 是控制平面之下的实际执行引擎。它不替代推理后端（SGLang、TensorRT-LLM、vLLM），而是将它们编排为协调的多节点推理系统。Dynamo 1.0 已于 GTC 2026 发布为生产级框架，在 NVIDIA Blackwell 硬件上实现了高达 7 倍的吞吐量提升。

①
SLO Planner（服务等级目标规划器）

监控多节点部署中的容量和 prefill 活动，动态调整 GPU 资源以持续满足服务等级目标（SLO）。通过考虑应用级 SLO（如 TTFT 和 ITL）以及 KV-Cache 传输成本，Planner 做出知情决策来重新分配 GPU 或改变推理策略以缓解瓶颈。

②
KV-aware Router（KV-Cache 感知路由器）

在大规模 GPU 集群的多节点部署中高效引导流量，最小化冗余 KV-Cache 重计算。核心算法基于 Radix Tree 和全局 KVIndexer（详见上一章节）。

③
NIXL 低延迟通信库

Dynamo 的专用通信库，用于 GPU 之间以及跨异构内存/存储层的高吞吐、低延迟数据传输。提供统一异步 API，智能抽象底层复杂性，支持 GPUDirect Storage (GDS)、UCX 等多种后端，自动选择最优数据路径。

④
KV Block Manager（KV 块管理器）

成本感知的 KV 缓存引擎，在不同内存层级间传输 KV-Cache：GPU HBM → CPU DRAM → 本地 SSD → 网络存储。释放昂贵的 GPU 内存同时维持用户体验。现已支持 S3/Azure Blob 存储后端和全局 KV 事件。

⑤
Disaggregated Prefill/Decode（解耦 Prefill/Decode）

将 LLM 推理的上下文处理（prefill）和生成（decode）阶段分离到不同 GPU 上，实现独立分配和优化。在 GB200 NVL72 上配合宽专家并行（wide expert parallel），SemiAnalysis InferenceX 基准测试显示性能提升 7 倍。

NVIDIA Dynamo 官方页面 + 技术博客
Dynamo 1.0 已交付成熟的生产级分布式推理框架，在主要行业和云平台上实现了经过验证的集成。最新进展包括：代理推理优化（优先级路由、缓存钉扎）、多模态加速（解耦编码/预填/解码，embedding 缓存——图像负载 TTFT 提升 30%）、原生视频生成支持、ModelExpress 通过检查点恢复和权重流实现 7 倍更快启动、零配置部署（DGDR）等。
nvidia.com/en-us/ai/dynamo/ · developer.nvidia.com/blog/nvidia-dynamo-1-production-ready/
06 — CMX 上下文记忆 + STX 存储架构
控制平面的「记忆层」：让 AI Agent 拥有跨会话的持久上下文

KV-Cache 感知路由解决了「当前会话内如何高效复用缓存」的问题。但 Agentic AI 面临一个更深层的挑战：多步推理、跨工具调用、跨会话的上下文记忆。当上下文窗口增长到百万级 token，传统 GPU HBM 容量成为瓶颈。这就是 CMX 和 STX 要解决的问题。

STX：存储成为推理流程的一部分

STX（Scalable Token eXchange）是 GTC 2026 发布的模块化参考架构，由 BlueField-4 DPU 加速。它的核心理念是：将存储纳入推理流程，而不仅仅在训练或数据准备阶段使用存储。Jensen Huang 在 GTC 上表示：「推理上下文需要一种全新的存储类别。STX 重新发明了存储栈。」

CMX：STX 的首个机架级实现

CMX（Context Memory eXtension）是 STX 架构的首个产品化实现，定位为「G3.5」存储层——介于本地节点存储（G3）和共享企业存储（G4）之间。

核心能力
CMX 技术特性

容量：每个 GPU Pod 提供 PB 级共享容量，使长上下文负载在 HBM 和 DRAM 驱逐后仍可保留历史。

预暂存：KV 块可从 CMX 预先移入 G2 或 G1 内存，在 decode 阶段前完成就位，减少解码器停滞和 GPU 空闲。

共享访问：上下文成为高带宽共享资源，编排器可跨 Agent 和服务协调访问，无需在每个节点独立重建。

硬件加速安全：BlueField-4 的加密和完整性加速器以线速处理 KV 流的加密和 CRC 校验，不增加主机 CPU 开销。BlueField-4 直接管理 NVMe SSD，在存储处理器层面保持上下文可访问，而非通过主机传输。

07 — 服务编排流程
从租户请求到推理执行的完整链路

NVIDIA 官方白皮书给出了完整的服务流程图，展示了企业租户、AI Grid Orchestrator 和终端用户服务之间的交互方式：

1
租户创建与部署请求
服务提供商或电信运营商管理租户创建。部署请求由每个租户发起，进入控制平面。
2
资源与意图引擎（Resource and Intent Engine）
控制平面通过集中式资源与意图引擎处理请求，将高层级需求翻译为基础设施操作。
3
决策与策略引擎（Decision and Policy Engine）
与 SDN 控制器协调进行网络创建，与 NVCF（NVIDIA Cloud Functions）协调实现可选的无服务器部署。
4
负载配置到 AI Grid 资源
将负载配置到 AI Grid 资源上。编排器维护目标状态——若节点失败，检测心跳丢失并在其他位置重新配置。
5
推理请求的实时路由
终端用户请求到达后，全局负载均衡器分发到活跃实例，LLM Router 进行语义分析和 KV-Cache 感知路由，Dynamo 执行实际的分布式推理。

08 — 基准验证数据
Comcast 实测：控制平面带来的可量化收益

控制平面的价值不是理论性的——Comcast 在 GTC 2026 上公布了基于 Personal AI 语音 SLM 的对比基准测试，直接展示了控制平面路由能力在集中式 vs 分布式架构下的实际差异。

NVIDIA 技术博客 · Comcast 基准测试
NVIDIA 技术博客详细报告：在所有测试场景中（从 P50 基线到 P99 突发），AI Grid 部署始终将语音交互的端到端延迟维持在 500ms 目标内。关键发现：集中式集群将大部分延迟预算消耗在 RTT（往返时间）上，因此必须以较低利用率运行以避免尾延迟违规。而 AI Grid 部署保持低 RTT，可以在相同延迟目标下安全地驱动 GPU 更高利用率——由此产生的每 Token 成本优势在突发时从 52.8% 扩大到 76.1%。

为什么控制平面是差距的关键

这组数据揭示了一个关键洞察：控制平面通过 KPI 感知路由将突发流量均匀分散到 4 个边缘节点（而非让单一集群过载），同时通过 KV-Cache 感知放置减少每个节点的冗余计算。结果是延迟更低、吞吐更高、每 Token 成本更低——这三个指标在集中式架构中是相互矛盾的（提高利用率会增加尾延迟），但在 AI Grid 控制平面下实现了同时优化。
