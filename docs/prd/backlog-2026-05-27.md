## 1. 模型网关（Model Gateway）

### 背景

当前 `backend/src/modules/ai-engine/llm/` 已含 25.7K LOC：

| 子能力                 | 现有位置                                    |
| ---------------------- | ------------------------------------------- |
| LLM 调用核心           | `ai-chat.service.ts` (2,794 LOC)            |
| HTTP 适配              | `ai-api-caller.service.ts` (2,016 LOC)      |
| 模型配置               | `ai-model-config.service.ts` (1,593 LOC)    |
| 连接测试               | `ai-connection-test.service.ts` (1,297 LOC) |
| 模型推荐               | `model-recommendation.service.ts` 等        |
| TaskProfile / 模型路由 | facade 层                                   |

"模型网关"通常涵盖以下职责，需用户确认哪些**新增**、哪些**从现有 llm 里抽出来重组**：

| 职责                                                 | 当前归属                               | 网关新增？ |
| ---------------------------------------------------- | -------------------------------------- | ---------- |
| Provider 路由（OpenAI / Anthropic / Grok / LiteLLM） | `ai-api-caller` 内嵌                   | ?          |
| 模型选择策略（按 modelType + TaskProfile）           | `ai-chat` 内嵌                         | ?          |
| BYOK（用户自带 key）                                 | `ai-app/byok/` 已有独立模块            | ?          |
| Failover / 降级                                      | 散落在 chat / caller                   | ?          |
| 成本归集 / 计费                                      | `ai-infra/credits/` + `BillingContext` | ?          |
| Rate limit / 配额                                    | `guardrails/` 部分                     | ?          |
| Streaming / SSE                                      | `ai-chat`                              | ?          |
| 模型黑名单 / 白名单                                  | scattered                              | ?          |

### 待澄清问题

启动前需要用户回答：

1. "网关"是**新建顶层模块**（`ai-engine/gateway/` 或 `ai-infra/model-gateway/`），还是把现有 `ai-engine/llm/` 重组？
2. **触发场景**：哪条调用链当前痛点最痛？（举例：grok 限流时 failover 失败？BYOK 成本归集错？跨 provider 模型名冲突？）

4. **与 BYOK 模块的关系**：BYOK 是网关的一种"密钥来源"，还是平行模块？

### 已知不要做的事

- 不要在没有具体使用场景前抽象出"网关 interface + 9 个 strategy 实现"——典型过度抽象（Karpathy 原则）
- 不要重命名已有 service 类（破坏 60+ 个调用方）

## 2. 快速构建 Agent 的 CLI

### 背景

`backend/src/modules/ai-app/agent-playground/` 已落地完整 mission 架构（30,832 LOC，14 个子目录）。**B1-B7 瘦身**完成后形态稳定，可作为 scaffold 模板。

新增一个 agent app 时，至少要复制：

| 模式                                                               | 来源                                        |
| ------------------------------------------------------------------ | ------------------------------------------- |
| `mission/agents/{role}/` 目录（含 SKILL.md / agent class / index） | playground 7 个 agent                       |
| `mission/roles/{role}.service.ts` invocation wrapper               | playground 7 个 role                        |
| `mission/projectors/`                                              | mission-view / todo-board / artifact 三件套 |
| `mission/pipeline/`                                                | dispatcher + stages                         |
| `mission/lifecycle/`                                               | store / event-buffer / config-snapshot      |
| `mission/query/`                                                   | canonical view query                        |
| `api/contracts/`                                                   | view-state contract                         |
| `api/controller/`                                                  | run / read controller                       |
| `events/`                                                          | event schemas                               |
| `module/`                                                          | nest module                                 |
| `runtime/`                                                         | gateway / config                            |

### 目标

```bash
# 一行命令生成最小骨架
npx genesis create:agent <agent-name> --stages s1,s2,s3 --domain <短描述>

# 输出目录：
backend/src/modules/ai-app/<agent-name>/
  mission/
    agents/<your-first-agent>/
      SKILL.md
      <your-first-agent>.agent.ts
      index.ts
    roles/<your-first-agent>.service.ts
    pipeline/
      <agent-name>-pipeline-dispatcher.service.ts
      stages/
        s1-foo.stage.ts
        s2-bar.stage.ts
        s3-baz.stage.ts
        radar-stage-types.ts  # rename to <agent-name>-stage-types.ts
    projectors/
      <agent-name>-mission-view.projector.ts
    lifecycle/
      <agent-name>-mission-store.service.ts
      <agent-name>-mission-event-buffer.service.ts
    query/
      <agent-name>-mission-query.service.ts
  api/
    contracts/view-state.contract.ts
    controller/<agent-name>-run.controller.ts
    dto/run-mission.dto.ts
  events/<agent-name>.events.ts
  module/<agent-name>.module.ts
  runtime/<agent-name>.config.ts

# 同时：
# - 把 module 注册到 app.module.ts
# - 生成 1 个最小 fixture spec
# - 生成 README.md 指南
```

### 约束（强制走 harness 不重造）

- agent class 走 `@/modules/ai-harness/facade` 的 `IAgent` / `AgentExecuteFn`
- stage runner 走 `ai-harness/runner` 标准接口
- event bus 走 `ai-harness/facade.DomainEventBus`
- 不允许在新 app 里写 agent loop / runner / event queue 这些已有的东西
- 不允许重复造 `runner-state.util` 等已上提到 harness 的工具

### 实现思路

- 用 **Plop.js** 或 **Nest CLI custom generator**
- Template 用 handlebars，从 playground 的实际文件抽取（避免漂移）
- 第一版做 dry-run 输出（打印 tree），用户 OK 后再写文件

### 待澄清问题

启动前需要用户回答：

1. **CLI 寄宿位置**：根目录 `scripts/create-agent.ts`？还是新建 `tools/genesis-cli/`？还是用 nest 自带 `nest g`？
2. **模板新鲜度**：playground 还在演进，模板要么手动抓快照（容易漂移），要么动态读 playground 文件（脆弱）。建议**抓快照 + 标注同步日期 + 在 CI 加 diff 看护**。
3. **MVP 范围**：只生成目录骨架就停？还是要自动 register module + 生成 1 个跑得动的 stage？
4. **测试模板**：每个 stage 自动生成 fixture spec，还是只生成空 spec 占位？
