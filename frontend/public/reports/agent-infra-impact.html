A.1Prefill 与 Decode 阶段的负载特征分裂

LLM 推理包含两个计算特征截然不同的阶段。Agent 场景让这种差异被极端放大：

维度Prefill（首 token）Decode（逐 token 生成）
计算特征Compute-bound
一次性处理全部 input tokens 的 Attention		Memory-bound
每步生成 1 token，主要瓶颈是读取 KV Cache
GPU 利用率>80% — 大规模矩阵乘法~30% — 内存读取主导，算力空闲
Agent 影响上下文持续增长，128K 上下文的 prefill 计算量 = O(n²)Agent 生成的每个 token 都要读取不断膨胀的 KV Cache
时间占比传统短请求约 30%Agent 长上下文下 decode 占比升至 >70%

核心矛盾：Agent 场景下 decode 阶段更长（生成更多 token），但 decode 的 GPU 利用率极低。大量 GPU FLOPS 在等待内存读取。

A.2Attention 计算复杂度与长上下文的乘数效应

Self-Attention 的计算量随序列长度二次增长，Agent 的长上下文将这个问题推向极端：

Attention FLOPs = 2 × n² × d × num_layers × num_heads
n = 序列长度, d = head dimension
4K 上下文1x传统单次调用基准
32K 上下文64xAgent 中等任务
128K 上下文1,024xAgent 复杂任务 / 多轮累积

Agent 每增加一轮对话，上下文增长，下一次 prefill 的计算量不是线性增加，而是二次增加。这也是为什么 Prefix Caching 的价值在 Agent 场景下被极度放大 — 缓存命中的部分跳过了这个 O(n²) 计算。

PAPERFlashAttention 将 Attention 的内存复杂度从 O(n²) 降为 O(n)，但计算量仍为 O(n²) — FlashAttention, NeurIPS 2022
PAPERRing Attention 将长序列分布到多设备以突破单卡显存限制 — Ring Attention, ICLR 2024
A.3GPU 利用率：从持续高负载到间歇性突发

传统推理请求密集到达，GPU 保持较高利用率。Agent 场景的 GPU 使用呈现波浪形：

阶段GPU 状态持续时间

优势：每次前向传播只激活部分专家（如 8 选 2），每 token 实际 FLOPs 低于同等参数量的 Dense 模型，适合 Agent 的高 token 消耗场景。

B.2Prefix Caching 的存储经济学

Agent 的多次调用共享大量相同 prefix（系统 prompt + 历史对话）。缓存这部分 KV 可以避免重复的 O(n²) prefill 计算：

场景共享 Prefix 占比缓存节省
同一 Agent 连续调用80-95%（系统 prompt + 累积历史不变）只需 prefill 本轮新增 input
多 Agent 共享系统 prompt5-30%（系统 prompt 相同）多个 Agent 共用同一份 KV Cache 前缀
RAG：共享知识库 chunk变化大（取决于检索重叠度）相同文档 chunk 的 KV 只计算一次

存储代价：缓存的 KV 需要额外存储空间。以 10 个 Agent 共享 8K prefix 为例，Llama 70B 的共享 KV 占 ~3 GB — 但如果不缓存，10 个 Agent 各自 prefill 8K tokens 的计算量 = 10 × O(8K²)，远大于 3 GB 的存储代价。

B.4写放大：Agent 迭代修正的存储成本

Agent 的核心行为是迭代修正 — 生成、评估、修改、重新生成。每次修正意味着：

• 旧版本的 KV Cache 失效，新版本需要重新计算和存储
• 对话历史追加新轮次，累积增长
• 工具调用结果可能需要更新（相同工具不同参数）
• 如果 Agent 回退决策（"这个方向错了，重来"），之前的 KV Cache 全部作废

在 coding agent 场景下尤为突出：SWE-Agent 在单个 bug 修复中平均产生 50-100 次 LLM 调用，每次调用都会修改代码状态和对话历史，存储层的写入量远大于最终输出量。

C.2KV Cache 跨节点迁移：推理场景的新流量大户

在解耦 prefill/decode 架构中，每次请求的 KV Cache 都需要从 prefill 节点传输到 decode 节点：

C.3Tensor Parallelism 通信：Agent 长上下文的 All-Reduce 压力

大模型推理通常需要 Tensor Parallelism（TP）将模型权重和计算分布到多张 GPU。每个 Transformer 层需要 2 次 All-Reduce 通信：

TP All-Reduce 数据量 / 层 = 2 × batch_size × seq_len × hidden_dim × precision

Agent 的长上下文（128K）使 prefill 阶段的 All-Reduce 数据量是 4K 上下文的 32 倍。这对 NVLink 带宽构成显著压力 — TP=8 时，每个 Transformer 层的通信量可达数百 MB。

Agent 特有影响：由于 Agent 上下文持续增长，后续轮次的 prefill 通信量越来越大。第一轮 4K 上下文 vs 第十轮 64K 上下文，TP 通信量相差 16 倍。调度器需要感知这个增长趋势。

起点 — 行为质变
Agent 让 LLM 从"被调用的函数"变成了"持续运行的进程"

传统 AI 调用是一个纯函数：输入 prompt，输出 response，无副作用、无状态、无后续。系统只需要管理"请求"。

Agent 打破了这个模型。一个 Agent 任务启动后，会自主循环：规划下一步行动 → 调用工具执行 → 观察执行结果 → 反思并决定是否继续。这个循环的终止条件不是系统决定的，而是 Agent 自己决定的 — 它可能循环 5 次，也可能循环 500 次。

130x

The Cost of Dynamic Reasoning, arXiv 2025
100 次
SWE-bench 不得不为 SWE-Agent 设置 100 次 LLM 调用上限，否则单个 bug 修复可能耗尽 200 万 token 预算。
SWE-bench Verified

这不是量变，是质变。系统管理的对象从"请求"变成了"进程"— 一个有状态的、持续运行的、自主决策终止条件的实体。这一个变化，驱动了后续所有基础设施层面的连锁反应。

如果只记住一句话：Agent 不是"更多的 API 调用"，而是一种全新的计算范式 — 从无状态函数到有状态进程。

第一性推论 — 核心资源重定义
上下文（KV Cache）取代算力，成为最稀缺的资源

Agent 循环意味着上下文只增不减：每一轮的系统 prompt、用户指令、工具调用结果、Agent 的推理过程，全部追加到对话历史中。第一轮可能是 4K tokens，到第十轮就膨胀到 64K，到第三十轮可能逼近 128K 上下文窗口的上限。

在 Transformer 架构下，这些上下文以 KV Cache 的形式驻留在 GPU 显存中。它的大小：

更关键的是 KV Cache 的生命周期发生了根本变化。传统推理中，KV Cache 的生命周期 = 单次请求的时长（毫秒到秒级），请求结束立即释放。Agent 场景下，KV Cache 的生命周期 = 整个任务的时长（分钟到小时级），而且中间还有工具调用的等待间隙 — Agent 在等搜索结果时 GPU 完全空闲，但 KV Cache 不能释放（否则恢复时要重新计算，128K 上下文的 prefill 计算量是 4K 的 1024 倍）。

KV Cache 就是 Agent 的"进程内存"。谁能高效管理 KV Cache 的生命周期，谁就掌握了 Agent 时代的算力经济学。

因果传导 — 三层基础设施同时破裂
计算、存储、网络不是三个独立问题，是"KV Cache 管理"的三个面

一旦理解了 KV Cache 是核心资源，就会发现传统推理基础设施的每一层假设都被打破了 — 而且这三层之间是紧密耦合的：

计算层的破裂：Attention 的计算量随序列长度二次增长（O(n²)）。Agent 的上下文持续膨胀意味着每一轮 prefill 的计算量都在增大。更糟糕的是，GPU 利用率呈现波浪形 — prefill 阶段 100% 满载（compute-bound），decode 阶段仅 ~30%（memory-bound），工具等待阶段 0%（完全空闲）。一个 Agent 任务中大部分时间 GPU 要么低效运转，要么空转。

36.9x
Orca 论文发现，传统 request-level batching 在这种负载模式下极度低效。改为 iteration-level scheduling 后吞吐提升 36.9 倍 — 说明旧调度模式浪费了 97% 以上的潜在吞吐。
Orca, OSDI 2022

存储层的破裂：KV Cache 在 GPU 显存（HBM）中放不下时，需要分层卸载到主机内存（DRAM，~200 GB/s）再到 SSD（~14 GB/s）。但 Agent 在工具等待后恢复时需要把 KV Cache 换回 GPU — 40 GB 的 KV Cache 通过 PCIe 5.0 换入需要 ~0.6 秒，这直接加到了用户感知的延迟上。传统推理不存在这个问题，因为 KV Cache 从不换出。

<4%
vLLM 的 PagedAttention 将 KV Cache 内存浪费从 60-80% 降至 4% 以下。其核心思想直接借鉴操作系统的虚拟内存分页 — 将 KV Cache 拆分为固定大小的"页"，按需分配和回收。
PagedAttention, SOSP 2023

网络层的破裂：为了解决计算层的 GPU 利用率问题，前沿架构将 prefill 和 decode 解耦到不同 GPU 池。但这意味着每次 Agent 循环都要通过网络传输 KV Cache — Llama 70B @ 128K 的 KV Cache = 40 GB，10 轮循环 = 400 GB 网络传输。100 个并发 Agent 就是 40 TB 的跨节点流量，这在传统推理中完全不存在。

1.56x
Helium 系统针对 Agent 工作流的跨调用依赖做了端到端优化，比 SOTA 推理系统提速 1.56 倍。其核心发现：现有系统（包括 vLLM）只优化单次调用，完全忽略了 Agent 多次调用间的依赖关系。
Helium, arXiv 2026
计算、存储、网络的压力来自同一个根源：Agent 让 KV Cache 变成了长寿命、大体积、需要迁移的资源。优化任何一层都会把瓶颈转移到另一层 — 这就是为什么需要系统性的架构范式转移，而不是局部优化。

Revenue = Tokens/Watt × Available GW

Deep Dive — 曲线为什么从左到右衰减？三面墙的物理根因

这条曲线的形状不是设计选择，是物理定律的直接结果。X 轴的"交互性"（TPS/User）本质上是 batch size 的反函数 — 越高的交互性要求越低的 batch size（更快响应 = 更少并发 = 更独占 GPU）。曲线下降的第一性原因是内存墙贯穿全程，互连墙和功率墙在中右区域叠加加剧：

Deep Dive 2 — 曲线上的两个拐点：两次计算范式的相变

仔细观察 Rubin 曲线，在 ~100 TPS/User 和 ~200 TPS/User 处各有一个明显的拐点。这不是设计选择，而是两次计算范式的相变：

拐点位置斜率物理成因为什么 Rubin 比 Blackwell 改善
拐点 1 ~100 TPS/User 陡峭
（效率断崖）		 Prefill → Decode 相变：低交互性时 GPU 做 Prefill（矩阵×矩阵，利用率 >80%，compute-bound）。交互性升到 ~100 TPS/User 时必须缩小 batch，系统切换到 Decode 主导（矩阵×向量，利用率骤降到 ~30%，memory-bound）。这是一个计算性质的突变 — 如同水在 100°C 突然变为蒸汽。 HBM4 22 TB/s（2.75× Blackwell）让 Decode 阶段的 HBM 带宽利用率不至于塌得那么快 → 断崖的起点被右推
拐点 2 ~200 TPS/User 变缓
（曲线趋平）		 Memory-bound → Power-bound 过渡：过了拐点 1 后已经是 Decode 主导。到 ~200 TPS/User（batch≈2-4）时，三个效应让曲线趋平：
• 静态功耗占比越来越大（HBM 刷新、NVLink 维持、漏电流是固定开销，不随 batch 变化 — 分母不变）
• 边际摊分递减（batch 从 32→16 损失巨大，从 4→2 损失小得多，已经很低了）
• HBM4 带宽过剩（batch=2 时实际 HBM 读取量远小于 22 TB/s，内存墙不再持续恶化）		 Blackwell 在 200+ TPS/User 仍在被内存墙压制（HBM3e 8 TB/s 不够）。Rubin 因 HBM4 的 2.75× 带宽余量，已在此区间"突破"内存墙，进入功率墙主导区域 — 功率墙的斜率天然比内存墙平缓
定量推导 — 为什么第二个拐点后曲线变平

以 Rubin 单 GPU Decode 阶段为例，推导 TPS/MW 的解析表达式。

// 参数定义（Rubin GPU, 70B FP4 模型） W = 35 GB // 模型权重 (70B × FP4) K = 0.64 GB // 单用户 KV Cache (70B @ 32K ctx, FP8) BW = 22,000 GB/s // HBM4 带宽 B = batch size // 并发用户数 P_s = 500 W // 静态功耗 (HBM 刷新 + 漏电 + NVLink) P_d = 1,800 W // 峰值动态功耗 u(B) = min(1, 2BK/W) // GPU 利用率 (Decode memory-bound) // 核心公式 T_decode = (W + B×K) / BW // 每步 Decode 时间 TPS/User = BW / (W + B×K) // X 轴：单用户 Token 速度 Total_TPS= B × BW / (W + B×K) // 系统总吞吐 P_total = P_s + P_d × u(B) // 总功耗（静态 + 动态） TPS/MW = Total_TPS / P_total // Y 轴 = [B × BW / (W+B×K)] / [P_s + P_d × u(B)]

关键：P_total 不是常数。当 B 减小时，GPU 利用率 u(B) 下降 → 动态功耗 P_d×u(B) 下降 → P_total 向 P_s（静态地板）收敛。分子（吞吐）和分母（功耗）同时下降，但下降速率不同 — 这就是曲线形状变化的数学根源。

定量总结：曲线形状由 TPS/MW = Total_TPS(B) / P_total(B) 的分子分母博弈决定。拐点 1（~100 TPS/User）是 Prefill→Decode 相变，分子断崖式下跌而分母不变 → 斜率最陡。拐点 2（~200-400 TPS/User）是 GPU 利用率 u(B) 快速下降的区间，分母（功耗）的下降速度追上了分子（吞吐）的下降速度 → 曲线趋平甚至出现局部峰值。B<32 后分母触及静态地板不再下降，曲线重新变陡。

核心洞察：两个拐点对应 Compute-bound → Memory-bound → Interconnect + Efficiency-bound 三段相变。第三段有两个叠加因素：(1) 互连墙 — 三级解耦引入逐级叠加的网络开销，NVLink 域内增加 ~33% 延迟、跨 Ethernet 时直接崩塌；(2) 能效墙 — GPU 静态功耗 + 网络设备功耗占 P_total 的 90%+，$/Token 失控。注意区分：Step 3 的"散热墙"是曲线左侧问题（少量 GPU 满载 2,300W → 物理热量需液冷）；曲线右侧是互连+能效的经济性问题。Groq LPU 打破能效墙（SRAM 无静态功耗），CPO/OCS 打破互连墙（降低网络功耗和延迟）— 两者配合才能有效"抬平"曲线右侧。

曲线优化技术：Speculative Decoding。除了硬件层面的突破，软件层面的 Speculative Decoding（投机解码）也在改变曲线形状：用小模型快速生成多个候选 Token（draft），再用大模型并行验证（verify）。这等效于在不降低质量的前提下将 decode 阶段从"逐 Token 串行"变为"批量并行验证"，在曲线上表现为同时"上移"（更高吞吐）和"右推"（更高交互性）。EAGLE-3 实现 2-3× decode 加速。Groq LPU 作为专用 decode 加速器，本质上是 Speculative Decoding 思路的硬件化实现 — 用专用低延迟芯片替代通用 GPU 做 Token 生成。

关键洞察：曲线下降的根因是内存墙（memory-bound decode + batch size 下降 → HBM 利用率下降），互连墙和功率墙在中右区域叠加加剧衰减。V6 的 Step 1-3 本质上就是在分析如何把这条曲线"上移"（提升 tokens/watt）和"右推"（在更高交互性下维持效率）。

技术启示
启示 1："推理上下文存储"是一个全新的硬件品类

ICMS 不是"给 SSD 换个名字"。传统存储的价值衡量标准是 IOPS 和吞吐量；推理上下文存储的价值衡量标准是 tokens/s 和 context restore latency。KV Cache 的访问模式是"大块顺序写 + 基于 Key 的随机读"，与传统 4KB 随机读写完全不同。

研究方向：SSD 控制器需要原生支持 NVMe KV 命令集和 KV Cache 访问模式优化；KV Cache 的预测性预取 — 基于 Agent 任务图提前将可能用到的上下文从 ICMS 加载到 HBM，减少 context restore stall。
启示 2：内存层次从 2 级变成 5 级，催生"内存编排"新赛道

传统是 SRAM → DRAM 两级。现在是 SRAM → HBM → Host DRAM → Local SSD / CXL Pool → ICMS/SSD 五级，每级容量差 10-100×，延迟差 3-10×。自动化地将 Agent 的工作集分布在 5 级内存层次中，是一个全新的系统优化问题，复杂度远超操作系统的虚拟内存管理。

研究方向：面向 LLM 推理的统一内存管理框架（LRU 不适用 KV Cache，需要 context-aware eviction）；跨会话 KV Cache 共享（多 Agent 处理同一文档时需 copy-on-write 机制）；内存层间数据迁移的性能建模。
启示 3：CXL 的真正价值不是"更多内存"，而是解耦计算和内存的扩展比

当前 GPU 服务器的内存是固定的（288GB HBM/GPU）。Agent 负载 100GB 时剩余浪费，500GB 时单卡不够。CXL 内存池化使内存成为可弹性分配的共享资源 — GPU 利用率有望从当前的 30-50% 大幅提升。

研究方向：面向 CXL Fabric 的 NUMA-aware 推理调度（KV Cache 在远端 CXL 内存的延迟需要调度器感知拓扑距离）；CXL 内存池的故障域设计 — 一个池节点故障影响范围和冗余级别研究。

Step 2 — 互连墙
推理流量重塑互连架构

互连墙由两股力量同时推动：一是上一步的内存池化（CXL/ICMS）需要跨节点带宽，二是 AI 集群本身的规模化 — 更大的模型需要更多 GPU 并行，MoE 架构需要 All-to-All 通信，集群从数百张卡扩展到数万张卡。当 NVLink 6 把单 GPU 的 Scale-up 带宽推到 3.6 TB/s 时，铜互连的信号衰减和功耗已经逼近物理极限。光互连不再是实验室产物，它在 2026 年正式进入大规模量产阶段。

三级解耦的网络含义：推理解耦越深，网络流量越接近甚至超过训练。Level 1 的 KV Cache 迁移是请求级的（低频大块），Level 2 的 Ping-pong 是 Token×Layer 级的（极高频小块），Level 3 的 All-to-All 是 Token×MoE层 级的。推理网络设计不能按"流量比训练小"来假设 — 解耦架构正在系统性地推高推理侧的网络需求。

为什么 CPO 不是可选项
三重约束叠加让 CPO 成为 Spine 层必选方案：
• 功率约束 — 102.4T 交换机如果用插拔式光模块，光互连功耗约占交换机总功耗的 40-50%。在机架功率已达 120-227 kW 的情况下，每节省 1W/端口都直接转化为可部署的额外 GPU 数量。Broadcom CPO 将光互连功耗降低 70%（3.5× 优于插拔式）。
• 端口密度约束 — 三级解耦（Prefill/Decode + Attention/FFN + MoE Expert）让推理集群的东西向端口需求成倍增长。插拔式光模块的前面板物理空间是硬上限。CPO 将光引擎集成到交换芯片封装内，绕过前面板密度瓶颈 — NVIDIA SN6800 实现 512×800G = 409.6T 单机，插拔式方案无法达到这个密度。
• 信号完整性约束 — 从交换芯片到前面板插拔模块的 PCB 走线在 200G/lane 速率下损耗已接近极限。CPO 将光电转换点从前面板移到芯片旁边（<10mm），彻底消除 PCB 高速走线损耗，是突破 200G/lane → 400G/lane 的工程前提。

Step 4 — 架构响应

映射到 Token Economics 曲线：NVIDIA 优化曲线左侧（高 batch、少 Agent、深度推理 — 单芯片爆发算力是核心）；Google 优化曲线右侧（低 batch、多 Agent、高交互性 — 低功耗和动态调度是核心）。谁对？取决于 Agent 工作负载的演进方向 — 如果 Agent 向"少而精"演进（复杂推理取代专家），NVIDIA 赢；如果向"多而快"演进（海量 Agent 取代客服），Google 赢。现实可能两者并存 — 这也是为什么 NVIDIA 收购 Groq（补"快"的短板），Google 开发 Ironwood（补"大规模推理"的短板）。

硬件为 Multi-Agent 量身定制的时代正式开始。NVIDIA 和 Google 给出了截然不同的答案 — 异构密度 vs 同构规模。这不只是技术路线之争，更是对 Agent 工作负载未来形态的不同赌注。核心判断：两种路线不会"谁取代谁"，而是在 Token Economics 曲线的不同区域各自占优。

重推理型（Rubin GPU / MI400）— 高算力+大 HBM，处理复杂 chain-of-thought；高吞吐型（Groq 3 LPU）— 极致 tokens/s/watt，片上 SRAM 为主，Agent-to-Agent 通信；平衡型（SambaNova SN50）— 多级内存热切换，灵活适配多模型混合流水线。

研究方向：异构芯片间任务调度器 — 如何在 GPU/LPU/RDU 之间动态分配 Agent 工作负载？哪些推理步骤适合 GPU（深度推理），哪些适合 LPU（快速响应）？这是一个全新的系统软件研究领域。Token 路由优化 — Multi-Agent 系统中 Token 流在异构芯片间的路由策略，类似网络 QoS 路由但粒度是 Token 级别。
启示 12：片上 SRAM 正在回归核心地位

Groq 3 LPU 500 MB SRAM / 150 TB/s 带宽；SambaNova SN50 432 MB SRAM。训练时代 SRAM 容量不值一提，但 Agent 推理时代，SRAM 的延迟优势（~1ns vs HBM ~10ns）对 Token 延迟极为关键。

研究方向：大容量 SRAM 的芯片架构设计回归；3D 堆叠 SRAM 的经济性分析；SRAM/HBM/DDR 多级缓存的 KV Cache 分层调度算法。
启示 13：TCO 模型需要从 $/FLOPS 转向 $/百万 Token

Groq LPU 报价 $45/百万 Token，Rubin 声称成本降 10×。传统的 $/FLOPS 评估方式无法区分这些根本不同的芯片类型。需要建立跨芯片类型的统一 TCO 对比方法论，核心指标是 tokens/watt 和 $/百万 Token（区分 prefill 和 decode 阶段）。

研究方向：构建 Agent 推理场景的标准化 benchmark（不只是吞吐量，还要包含多步推理延迟、KV Cache 换入换出开销、Multi-Agent 协调开销）。
启示 14：AI 工厂需要新的"基础设施操作系统"

当前管理碎片化：Kubernetes/Slurm 调度 GPU，vLLM 管理 KV Cache，NCCL 管理通信，BMC 管理散热。在 NVL72 + CXL 池 + ICMS + 光互连的架构下，这些组件必须作为一个整体来管理。NVIDIA Dynamo 是早期尝试，但远远不够。

研究方向：统一资源抽象 — 将 GPU 算力、HBM、CXL 内存、ICMS 存储、NVLink/光互连带宽统一为可调度资源；推理感知调度 — 传统调度器不理解 prefill vs decode、KV Cache 亲和性、MoE 路由；多租户 AI 工厂 — 一个 NVL72 同时服务多个客户时的 KV Cache 隔离和 SLA 保证。这是最大的系统软件机遇，相当于 AI 数据中心的 Linux 内核。

启示 15：开放标准 vs 封闭生态的决战时间表已明确

NVIDIA 通过 NVLink + CUDA 构建的封闭生态正在四个方向同时遭遇开放替代的挑战：

• Scale-up 互连：NVLink 6 vs UALink 2.0 + OCI MSA（2026 Q4 硬件可用性是关键）
• 内存互连：NVIDIA 私有 vs CXL 4.0（GPU 原生支持程度决定采纳速度）
• 推理芯片：Rubin + Groq vs AMD MI400 + SambaNova + Google TPU（Agent 推理 TCO 竞争）
• 机架标准：DGX/HGX vs OCP Catalina / Mount Diablo（供应链灵活性）

研究方向：如果 UALink + OCI 能在性能上达到 NVLink 的 80%+，将显著改变 NVIDIA 的市场定价权和客户锁定能力。2026 Q4 至 2027 H1 是关键窗口 — OCI Gen1 规范完成 + 首批 UALink 交换机出货，届时客户将首次有实质性的"第二选择"。
研究优先级排序（按产业确定性 × 投资回报窗口）

(1) 光互连芯片（OCI MSA 生态）— 开放标准带来的从属价值链重构，确定性最高，窗口 2026-2028
(2) 液冷散热（Eaton/Boyd、Vertiv、Modine）— 全液冷强制标配带来的规模跃升，Vera Rubin 无风冷版本是最硬的需求信号

(4) 电力基础设施（核电 SMR、800V DC PDU）— 长周期但极高确定性，数据中心选址的硬约束

核心判断：每一条研究线索都可以追溯到同一个源头 — Agent 让 KV Cache 变成了长寿命、大体积、需要跨节点迁移的资源。这个源头变化是所有下游技术判断的锚点。

Forward-Looking — 有立场的预判（2026 Q3 - 2028）

以下预判基于本文数据推演，带有明确的条件假设和时间锚点。

预判 1：三标准不会决出唯一赢家，而是分层共存（2027 定型）

NVLink 继续统治 Scale-up 最内层（机架内 GPU-to-GPU），因为 NVSwitch ASIC + CUDA/NCCL 的软硬件协同优势短期无法撼动。ESUN 赢得 Scale-up 外层和 Scale-out（机架间 Ethernet），因为 Broadcom TH6 + Arista 系统的 OEM 覆盖率远超 NVIDIA Spectrum-X。UALink 成为 AMD/Intel 生态的 Scale-up 方案，但规模有限 — 交换机要到 Q4 2026 才出货，错过了 Blackwell/Rubin 第一波部署窗口。

条件：如果 UALink 交换机延迟到 2027 H1，ESUN 将进一步吃掉 UALink 的地盘（AMD 已经在用 UALink over Ethernet 作为过渡方案）。观测指标：Upscale AI 和 Astera Labs 的 UALink 交换机出货时间 — 这是整个预判的枢轴变量。

预判 2：光互连在 Scale-up 层的渗透不晚于 2028（功率门控触发）

NVL72 内部 5,000+ 铜缆 + 20 kW 光功耗惩罚 = 当前铜仍然赢。但 Broadcom CPO 已实现 70% 功耗降低，且 Vera Rubin Ultra（2027, 600 kW/rack）的规模将让铜缆的物理极限（信号完整性、线缆密度、散热）同时触及。OCI MSA Gen1 规范（2026 Q4）+ Feynman 光学 NVLink（2028 路线图）= 光在 Scale-up 层的渗透窗口已锁定在 2027-2028。

条件：CPO 良率和成本如果不能在 2027 年达到铜互连的 1.5 倍以内，渗透会推迟到 2029。观测指标：Broadcom TH6-Davisson CPO 和 NVIDIA Spectrum-X Photonics 的实际出货量和客户反馈。

预判 3：训练和推理数据中心将在 2027-2028 年开始物理分离

功率行为差异（18MW/2s vs 4MW 摆幅）、选址约束差异（偏远 vs metro）、可中断性差异（可暂停 vs 不可暂停）、GPU 利用率差异（90% vs <50%）— 四个维度的结构性差异叠加，让"一座数据中心同时做训练和推理"成为次优方案。Meta 已经用 MTIA 芯片做推理、GPU 做训练，实现了芯片级的分离；下一步是设施级的分离。

谁先动：Meta 已实现芯片级分离（MTIA 推理 + GPU 训练），Google 已实施"电网压力时暂停 AI 工作负载"策略 — 这两家最可能率先实现设施级分离。条件：如果推理硬件功率密度追上训练，物理分离动力会减弱，但选址和可中断性差异仍然存在。观测指标：Hyperscaler 新建数据中心的用途标注是否开始分化（training-optimized vs inference-optimized）。

预判 4：OCS（光电路交换）将成为推理数据中心的标配网络层（2028）

推理集群需要为不同模型动态分配 GPU 池（DeepSeek 用 64 GPU，Llama 用 8 GPU，按需绑定）。传统固定拓扑需要物理换线或预配所有路径。OCS 的电路重配置能力（μs-ms 级切换，零功耗维持）天然匹配这一需求。Google 已部署数万端口 OCS 并验证了 40% 功耗降低和 30% 成本降低。当推理成为主导负载，OCS 的价值从"nice-to-have"变成"must-have"。

商业化信号已出现：Lumentum OCS 营收指引 2026 年底达 ~$1 亿/季（约占市场一半）；OCP 成立 OCS 标准化项目（Google/Microsoft/NVIDIA 参与）。条件：商用 OCS 的端口规模和交换延迟能否匹配 Google 自研水平。观测指标：首批非 Google 的 OCS 大规模部署何时出现（Meta 和 Microsoft 是最可能的早期采用者）；Cignal AI $25 亿（2029）市场预测是否兑现。

预判 5：MoE 推理流量将推动推理网络带宽向训练网络看齐（2026 H2 - 2027）

当前推理网络设计普遍假设"推理流量小于训练，可以超配"。但 MoE >60% 采用率 + 每层两次 All-to-All 通信 + DeepSeek-V3 计算通信比 1:1 = 推理流量已经接近训练量级。2026 H2 当 Vera Rubin NVL72 开始大规模部署 MoE 推理时，基于"推理可降配"假设设计的网络将首先暴露带宽不足。这将倒逼推理网络的 Leaf→Spine 超配比从 2:1~4:1 回收到 1:1~2:1。

条件：如果 Dense 模型回潮（MoE 占比下降），推理网络可以维持超配。观测指标：2026 H2 - 2027 H1 新部署的推理集群中，Leaf→Spine 的实际超配比是否从 4:1 收紧。

预判 6：Token 效率曲线将在 2027-2028 发生形状质变 — 从"GPU 陡降形"到"异构平缓形"

当前曲线（TPS/MW vs TPS/User）呈"左高右陡降"的 GPU 特征形状 — 内存墙和功率墙在低 batch 区域同时打击 tokens/watt。两股力量将改变曲线形状：软件侧，Speculative Decoding 将 decode 从逐 Token 串行变为批量并行验证（EAGLE-3 2-3× 加速），等效于曲线全域上移；硬件侧，Groq LPU（SRAM 无 HBM 空转功耗）+ CPO（Spine 层 tokens/watt 提升 3.5×）+ OCS（动态拓扑消除布线浪费），三者叠加将显著抬平曲线右侧。

条件：Groq LPU 量产规模和生态成熟度（当前仅在 NVIDIA 平台内验证）。观测指标：2027 H2 GTC 上黄仁勋是否展示形状明显不同的新曲线 — 如果右侧衰减斜率从 ~45° 降到 ~20°，就是形状质变的信号。

预判 7：CXL 内存池化将在 2028 年达到推理集群 10%+ 渗透率 — 软件生态是门控

CXL 硬件不是瓶颈（服务器渗透率 88%、Marvell CXL 交换芯片已出货）。真正的门控是软件生态：推理框架（vLLM/SGLang/Dynamo）需原生感知 CXL 内存池、KV Cache 调度器需理解 CXL 延迟特征（比 HBM 高但比 SSD 低）、多租户需要新的内存保护机制。当软件层就绪后，CXL 将从"硬件可用但无人用"快速切换到"推理集群标配"。

条件：如果 NVIDIA 在 CUDA/Dynamo 中原生支持 CXL 池化，采纳大幅加快；如果不支持（为推 NVLink 生态），CXL 池化主要在 AMD/Intel 平台发展，渗透率受限于非 NVIDIA 市场份额。观测指标：vLLM 或 SGLang 是否在 2026 H2 - 2027 H1 发布 CXL-aware KV Cache 管理功能。

2025 H2 到 GTC/OFC 2026 的核心变化不是"更大更快"，而是：

瓶颈从算力转向内存，催生 ICMS 和 CXL 内存池化；
Scale-up 互连三标准混战，ESUN/UALink 打破 NVLink 20 年垄断；
MoE 模糊训练与推理的网络边界，推翻"推理可降配"的假设；
异构芯片取代 GPU 单一架构，7 芯片平台定义新范式；
训练与推理的功率行为截然不同，走向物理分离。

推理作为一种与训练根本不同的工作负载，
正在重新定义算力底座的每一层设计假设。

CH.02 Agent 行为冲击：长程占用、跨步骤状态保持、多 Agent 通信、全时运行，四重负载特征冲击现有基础设施 行为冲击

CH.03 Agent 负载解析：KV Cache 线性膨胀、跨节点调度倍数放大、峰值变常态、Prefill/Decode 共用硬件致热电双重浪费 负载解析
四重行为冲击的技术根因逐一浮现。根因一：Agent 长上下文驱动 KV Cache 随序列长度线性膨胀，70B 模型 128K 上下文下单请求 KV Cache 占用约 40GB，一块 H100（80GB）被一个用户吃掉一半——这是内存墙的技术根因。根因二：多 Agent 并发下，P/D 分离架构中 KV Cache 跨节点传输量达每请求数百 MB——这是互联墙的技术根因。根因三：Agent 全时运行将负载从间歇性峰值变为 7×24 持续满载——这是散热墙的技术根因。根因四：Prefill 阶段计算密集，Decode 阶段访存密集，共用同一套高功耗 GPU，Decode 阶段算力大部分闲置但功耗不减——热白散了，电白烧了，这是散热墙与能效墙的共同技术根因。

CORE KV Cache Key-Value Cache

一句话：KV Cache 是 Transformer 推理中用存储换计算的核心机制 — 缓存历史 token 的 Key 和 Value 向量，避免每生成一个新 token 都从头重算。

为什么需要它：LLM 生成文本是自回归的，一次只生成一个 token。每个新 token 在计算 Attention 时需要"看到"前面所有 token 的 K 和 V。如果不缓存，每步都要重新计算全部历史 token 的 K、V — 总计算量是 O(1+2+...+n) = O(n²)。有了 Cache，每步只计算新增那一个 token 的 K 和 V，历史的全部从 Cache 读取。

存了什么：模型每一层（L 层）、每一组 KV Head（H_kv 组）对每一个 token 计算出的 K 向量和 V 向量。每生成一个新 token，在每一层追加一行 K 和一行 V。

• 大小膨胀 — 上下文持续累积，第一轮 4K → 第二十轮 80K，Cache 从 1.2 GB 涨到 25 GB
• 寿命拉长 — 从秒级（传统调用）变成分钟/小时级（Agent 任务全程）
• 中间空闲 — 工具等待期间 GPU 空闲，但 Cache 不能释放（否则恢复代价 = 重新 prefill 全部历史，计算量是 O(n²)）

KV Cache 就是 Agent 的"进程内存"。谁能高效管理 KV Cache 的生命周期（分配、共享、换入换出、释放），谁就掌握了 Agent 时代的算力经济学。
CORE Prefill / Decode 两阶段 Prefill Phase & Decode Phase

LLM 推理天然分成两个计算特征截然不同的阶段：

Prefill（首 token 生成前）：处理用户输入的全部 prompt tokens。所有 token 并行计算 K、V、Q，一次性填充 KV Cache。Compute-bound — GPU 算力满载（大规模矩阵乘法），利用率 >80%。

Decode（逐 token 生成）：每步只处理一个新 token。新 token 的 Q 与 Cache 中全部 K 做点积 → 权重 × 全部 V → 输出。然后新 token 的 K、V 追加到 Cache。Memory-bound — 计算量小，主要时间在从 HBM 读取 KV Cache，GPU 利用率仅 ~30%。

CORE Self-Attention 自注意力机制

Transformer 的核心计算。每个 token 被线性变换为三个向量：

Q (Query) — "我在找什么" ← 当前 token 用来查询历史
K (Key) — "我能提供什么" ← 历史 token 的索引
V (Value) — "我的实际内容" ← 历史 token 的数据

计算流程：

// 1. 相关性打分 Scores = Q × K⊃T / √d // [1, n] — 当前 token 对每个历史 token 的关注度 // 2. 归一化 Weights = softmax(Scores) // [1, n] — 权重和为 1 // 3. 加权聚合 Output = Weights × V // [1, d] — 融合了历史信息的输出向量

复杂度：Attention 计算量随序列长度 二次增长（O(n² × d)）。Agent 的上下文从 4K 增长到 128K 时，Attention 计算量增长 1024 倍。

这就是为什么 Prefix Caching 的价值在 Agent 场景下被极度放大 — 缓存命中的部分跳过了整个 O(n²) 的 Attention 计算。

OPT Prefix Caching 前缀缓存 / RadixAttention

问题：Agent 多次调用共享大量相同前缀（系统 prompt + 历史对话），每次都重新 Prefill = 重复 O(n²) 计算。

方案：缓存共享前缀的 KV Cache，后续调用直接复用，只 Prefill 新增部分。

// Agent 第 1 次调用 Prefill [系统prompt 2K + 用户指令 1K] → KV Cache 全部计算并缓存 // Agent 第 2 次调用 Cache 命中 [系统prompt 2K] → 跳过! 只需 Prefill [新增 1K] 计算量从 O(3K²) 降为 O(1K²)，节省 89% 的 Prefill 计算 // 多 Agent 共享同一系统 prompt Agent A、B、C 共用同一份 prefix KV Cache → 收益 3 倍放大

ARCH Tensor Parallelism 张量并行 (TP)

将模型权重按列或行维度切分到多张 GPU 上，每张 GPU 计算部分结果，然后通过 All-Reduce 通信合并。

大模型推理（如 70B）单卡放不下，必须使用 TP。每个 Transformer 层需要 2 次 All-Reduce（Attention 和 FFN 各一次），通信量与序列长度成正比。

TP All-Reduce 数据量 / 层 = 2 × batch × seq_len × hidden_dim × precision

Agent 影响：上下文从 4K 增长到 128K 时，Prefill 阶段的 All-Reduce 数据量增长 32 倍，对 NVLink 带宽（900 GB/s）构成压力。而且 Agent 每轮上下文更长，后续轮次的 TP 通信越来越重。

为什么 LPU 能在曲线右侧"救场"：

• 零 HBM 静态功耗：SRAM 不读不耗电。GPU 在 B=4 时有 500W 静态功耗（占 90%），LPU 几乎没有 — tokens/watt 在小 batch 下远优于 GPU
