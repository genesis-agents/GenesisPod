# 完成验证技能

> 在声明完成前，确保修复/实现真正解决了问题。

## 核心原则

### 完成的定义

**不是**完成：

- 代码写完了
- 类型检查通过了
- "应该可以了"

**真正**完成：

- 问题确实解决了
- 测试验证通过了
- 边界情况处理了
- 可以复现验证

## 验证清单

### 1. Bug 修复验证

```markdown
## Bug 修复验证

### 原始问题

- 问题描述: [...]
- 复现步骤: [...]
- 期望行为: [...]
- 实际行为: [...]

### 修复验证

- [ ] 能复现原始问题（修复前）
- [ ] 修复后问题不再出现
- [ ] 相关功能未受影响
- [ ] 边界情况已考虑
- [ ] 相关测试通过

### 回归检查

- [ ] 相关模块功能正常
- [ ] 全局测试无新增失败
- [ ] 性能无明显下降
```

### 2. 功能实现验证

```markdown
## 功能实现验证

### 功能需求

- 需求描述: [...]
- 验收标准: [...]

### 实现验证

- [ ] 核心功能正常工作
- [ ] UI 符合设计稿
- [ ] API 符合接口定义
- [ ] 错误处理完善
- [ ] 边界情况处理

### 测试覆盖

- [ ] 单元测试覆盖主路径
- [ ] 边界条件有测试
- [ ] 错误路径有测试
```

### 3. 重构验证

```markdown
## 重构验证

### 重构目标

- 原因: [...]
- 预期改进: [...]

### 行为不变验证

- [ ] 所有现有测试通过
- [ ] 功能行为无变化
- [ ] API 兼容性保持
- [ ] 性能无退化

### 质量提升验证

- [ ] 代码可读性提升
- [ ] 复杂度降低
- [ ] 重复代码减少
```

## 验证方法

### 方法 1: 测试验证

```bash
# 运行相关测试
npm test -- --grep "相关测试名"

# 运行覆盖率检查
npm run test:coverage -- --collectCoverageFrom="src/modules/xxx/**"
```

### 方法 2: 手动验证

```bash
# 启动开发服务
npm run dev

# 按复现步骤验证
# 1. 打开页面 X
# 2. 执行操作 Y
# 3. 验证结果 Z
```

### 方法 3: 自动化验证脚本

```bash
#!/bin/bash
# verify-fix.sh

echo "🔍 验证修复..."

# 1. 类型检查
npm run type-check || { echo "❌ 类型检查失败"; exit 1; }

# 2. 单元测试
npm test || { echo "❌ 测试失败"; exit 1; }

# 3. 相关功能测试
npm test -- --grep "相关模块" || { echo "❌ 相关测试失败"; exit 1; }

echo "✅ 验证通过"
```

## 常见遗漏

### 遗漏 1: 只验证正常路径

```
❌ 只测试 "正确输入 → 正确输出"
✅ 还要测试:
   - 空输入
   - 无效输入
   - 边界值
   - 并发情况
```

### 遗漏 2: 只验证单一位置

```
❌ 只验证改动的文件
✅ 还要验证:
   - 调用方
   - 依赖方
   - 相似功能
```

### 遗漏 3: 只验证开发环境

```
❌ 只在本地验证
✅ 还要考虑:
   - 不同浏览器
   - 不同数据量
   - 生产环境配置
```

## 验证命令

```bash
# 完整验证
/verify-completion

# 快速验证（只跑测试）
/verify-completion --quick

# 详细验证（含性能测试）
/verify-completion --full
```

**记住**: 没有验证的完成不是真正的完成。多花 2 分钟验证，少花 2 小时返工。
