# 计划编写技能

> 将工作分解为可管理的小任务，每个任务完整且可独立执行。

## 核心原则

### 1. 任务粒度

每个任务应该：

- **时间**: 2-5 分钟可完成
- **范围**: 单一职责，聚焦一件事
- **输出**: 明确的交付物
- **验证**: 清晰的完成标准

### 2. 规格完整性

每个任务必须包含：

```markdown
## 任务 N: [任务名称]

### 目标

[一句话描述要达成什么]

### 输入

- [需要什么前置条件或数据]

### 步骤

1. [具体操作步骤1]
2. [具体操作步骤2]
   ...

### 输出

- [具体交付物是什么]

### 验证

- [ ] [如何验证完成了]
```

## 计划模板

```markdown
# [功能/任务名称] 实现计划

## 背景

[为什么要做这个，解决什么问题]

## 目标

[最终要达成什么效果]

## 任务分解

### 任务 1: [准备工作]

[按模板填写]

### 任务 2: [核心实现]

[按模板填写]

### 任务 3: [测试验证]

[按模板填写]

## 依赖关系

- 任务 2 依赖任务 1
- 任务 3 依赖任务 2

## 风险点

- [可能遇到的问题和应对方案]

## 检查点

- [ ] 任务 1 完成后：检查 X
- [ ] 任务 2 完成后：检查 Y
```

## 分解策略

### 按层次分解

```
UI 层 → API 层 → Service 层 → 数据层
  ↓       ↓         ↓          ↓
任务1   任务2     任务3      任务4
```

### 按功能分解

```
功能 A    功能 B    功能 C
   ↓         ↓         ↓
任务1-3   任务4-6   任务7-9
```

### 按阶段分解

```
准备 → 实现 → 测试 → 集成 → 验收
```

## 常见反模式

| 反模式   | 问题           | 修正                               |
| -------- | -------------- | ---------------------------------- |
| 任务过大 | "实现用户模块" | 拆分为：模型、服务、控制器、测试   |
| 任务模糊 | "优化性能"     | 明确：识别瓶颈、优化查询、添加缓存 |
| 缺少验证 | 没有完成标准   | 添加：测试通过、代码审查通过       |
| 跳过依赖 | 假设前置已完成 | 明确列出依赖和前置条件             |

## 输出格式

使用此技能后，将在当前目录生成 `PLAN.md` 文件，包含：

1. 完整的任务分解
2. 每个任务的详细规格
3. 依赖关系图
4. 检查点列表
5. 风险评估

---

**记住**: 好的计划是成功的一半。花时间在计划上，执行时事半功倍。
