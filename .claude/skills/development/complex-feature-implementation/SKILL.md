# 复杂功能实现验证技能

> 确保复杂功能实现完整、正确，避免"写完即忘"的问题

---

## 适用场景

- 涉及多个服务/模块的功能
- 有数据转换、合并逻辑的功能
- 新增数据库模型的功能
- 需要初始化或注册的功能

---

## 核心原则

```
实现 → 连接 → 验证 → 模拟
  ↓      ↓      ↓      ↓
写方法  追踪调用  类型检查  数据追踪
```

**不是**：写完 → 类型检查通过 → 完成

---

## 强制验证清单

### 1. 方法级验证（每写完一个方法）

```
□ 谁调用这个方法？
  → grep -r "methodName(" 确认调用点

□ 调用点参数完整吗？
  → 对比方法签名和实际调用

□ 返回值被正确使用了吗？
  → 检查调用方是否处理了返回值
```

### 2. 服务级验证（每写完一个服务）

```
□ Module 注册
  → providers: [NewService]
  → exports: [NewService]（如果其他模块需要用）

□ 依赖注入
  → constructor(private readonly newService: NewService)

□ 初始化调用
  → 需要启动时执行的方法在 onModuleInit 中调用了吗？

□ 错误处理
  → 初始化失败会导致系统崩溃吗？需要 try-catch 保护吗？
```

### 3. 数据合并验证（涉及 merge/combine 逻辑时）

```
□ 浅合并 vs 深合并
  → { ...a, ...b } 是浅合并，嵌套对象会被完全覆盖
  → 需要递归合并时必须实现 deepMerge

□ 数组处理
  → 覆盖？追加？去重追加？

□ 手动模拟数据
  → 准备 2-3 组测试数据
  → 手动追踪合并过程
  → 验证结果符合预期
```

**示例**：

```typescript
// 浅合并陷阱
const a = { nested: { x: 1, y: 2 } };
const b = { nested: { x: 10 } };
const result = { ...a, ...b };
// result.nested = { x: 10 }  ← y 丢失了！

// 正确做法：深合并
deepMerge(a, b);
// result.nested = { x: 10, y: 2 }  ← y 保留
```

### 4. 类型安全验证（涉及 JSON/any 时）

```
□ 类型断言 (as T) 是危险的
  → 编译器不检查运行时数据
  → 必须有默认值兜底

□ JSON 字段访问
  → 用可选链 data?.field?.subfield
  → 或先合并默认值再使用
```

**示例**：

```typescript
// 危险：假设数据结构正确
const rules = jsonData as DialogueRules;
rules.techniques.join(); // 如果 techniques 是 undefined 就崩溃

// 安全：先合并默认值
const rules = { ...DEFAULT_RULES, ...jsonData };
rules.techniques.join(); // 即使 jsonData 缺字段也安全
```

### 5. 端到端验证（功能完成后）

```
□ 画出完整数据流
  用户操作 → API → Service → Database → Response → UI

□ 追踪关键数据
  在每个节点，数据长什么样？

□ 边界情况
  - 数据为空
  - 数据缺失
  - 关联数据被删除
  - 首次使用（无历史数据）
```

## 常见陷阱速查

| 陷阱         | 症状                  | 检查方法                | 修复                   |
| ------------ | --------------------- | ----------------------- | ---------------------- |
| 调用点遗漏   | 方法存在但没效果      | `grep -r "methodName("` | 更新调用点             |
| 初始化遗漏   | 服务正常但数据空      | 检查 `onModuleInit`     | 添加初始化调用         |
| 浅合并覆盖   | 部分配置丢失          | 打印合并前后数据        | 改用深合并             |
| 类型断言崩溃 | 运行时 undefined 错误 | 检查 JSON 字段访问      | 添加默认值             |
| 模块未注册   | 注入失败错误          | 检查 `providers`        | 添加注册               |
| 循环依赖     | 启动失败              | 检查 import 关系        | 用 `forwardRef` 或重构 |

## 验证流程模板

```markdown
## [功能名] 实现验证

### 1. 组件清单

- [ ] Service: XxxService
- [ ] Module 注册
- [ ] 数据库模型（如有）

### 2. 调用链验证

- [ ] 入口点 → 方法A → 方法B → 出口点
- [ ] 每个箭头的参数传递正确

### 3. 数据流模拟

输入：xxx
↓
处理：xxx
↓
输出：xxx（符合预期？）

### 4. 边界情况

- [ ] 输入为空
- [ ] 部分数据缺失
- [ ] 首次使用
```
