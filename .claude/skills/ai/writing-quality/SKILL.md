# AI 写作质量增强技能

> 用于评估和提升 AI 生成小说/长文的质量，解决重复、节奏、风格等常见问题。

---

## 适用场景

- 评估 AI 生成的小说章节质量
- 诊断写作质量问题（重复、节奏慢、风格不一致）
- 优化写作系统的 prompt 和约束

---

## 质量评估维度

### 1. 表达多样性 (Diversity)

| 检查项   | 问题表现                         | 解决方案           |
| -------- | -------------------------------- | ------------------ |
| 词汇重复 | "心中一震"、"微微一笑"等高频出现 | 使用表达冷却系统   |
| 句式单一 | 每句长度相似，缺乏变化           | 强制句式多样性约束 |
| 开场雷同 | 每章都是"站在xxx中"开始          | 章节开场类型冷却   |

**检测命令**：

```bash
# 检测高频表达
grep -c "心中一震\|心头一紧\|微微一笑" novel.md
```

### 2. 叙事节奏 (Pacing)

| 检查项   | 问题表现                   | 解决方案             |
| -------- | -------------------------- | -------------------- |
| 主角被动 | 连续多章主角只观察不行动   | 强制主角行动约束     |
| 情节停滞 | 没有实质性进展             | 每章必须有一个推进点 |
| 节奏单一 | 要么全是冲突，要么全是铺垫 | 张弛有度约束         |

**检测关键词**：

- 主动行动词：决定、选择、采取、反击、计划
- 被动观察词：只能看着、默默注视、无能为力

### 3. 角色一致性 (Character Consistency)

| 检查项   | 问题表现             | 解决方案         |
| -------- | -------------------- | ---------------- |
| 语气雷同 | 所有角色说话方式相同 | 角色人格约束系统 |
| 性格偏离 | 角色行为与设定不符   | 角色档案检查     |
| 称谓混乱 | 自称/敬称使用错误    | 身份语言规范     |

### 4. 风格契合度 (Style Match)

| 风格   | 核心特征           | 常见问题     |
| ------ | ------------------ | ------------ |
| 甄嬛传 | 一语双关、暗藏机锋 | 对话太直白   |
| 金庸   | 大气磅礴、侠义豪情 | 缺乏武侠氛围 |
| 古龙   | 短句犀利、氛围神秘 | 句子太长     |

---

## 质量评分标准

```
综合评分 = 多样性×30% + 角色一致性×25% + 情节推进×20% + 风格契合×25%
```

| 评分 | 等级   | 说明             |
| ---- | ------ | ---------------- |
| 8-10 | 优秀   | 可直接发布       |
| 6-8  | 良好   | 需要小幅修改     |
| 4-6  | 及格   | 需要重写部分章节 |
| <4   | 不及格 | 需要重新规划     |

## 常见问题诊断流程

### 问题1：章节开场重复

**症状**：多章以相同模式开场（如"站在庭院中"）

**诊断步骤**：

1. 提取每章前 100 字
2. 对比开场模式
3. 检查表达冷却系统是否生效

**解决方案**：

- 扩展 `CHAPTER_OPENING` 类型的检测模式
- 增加冷却章节数（建议 25 章）
- 在 prompt 中明确要求开场多样化

### 问题2：主角过于被动

**症状**：主角连续多章只观察、等待、被动应对

**诊断步骤**：

1. 统计每章主角的"主动行动"次数
2. 检测被动关键词密度
3. 评估连续被动章节数

**解决方案**：

- 使用 `NarrativePacingService` 强制主角行动约束
- 在章节大纲中预设主角决策点
- 限制连续被动章节数（建议 ≤2）

### 问题3：对话风格不符

**症状**：宫斗剧对话太直白，缺乏暗语机锋

**诊断步骤**：

1. 检查是否使用了正确的风格预设
2. 评估对话中的"话中有话"比例
3. 对比不同角色的语气差异

**解决方案**：

- 使用优化后的 `web_gongdou` 风格预设
- 在 prompt 中加入具体示例
- 使用角色人格约束系统区分语气

## 三层风格配置架构

> 解决"甄嬛传级别标准"硬编码问题，实现可配置的风格系统

### 架构设计

```
┌─────────────────────────────────────────┐
│  代码层 (Code Layer)                      │
│  - 基础风格特征（不可变）                   │
│  - writing-style-presets.ts              │
│  - 如：web_gongdou 基础节奏、对话风格       │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  数据库层 (Database Layer)                │
│  - WritingStyleTemplate 模型              │
│  - 详细规则、示例、避免模式（可配置）        │
│  - 系统预设 + 用户自定义                   │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  项目层 (Project Layer)                   │
│  - WritingProject.styleOverrides         │
│  - 项目特定覆盖（优先级最高）              │
└─────────────────────────────────────────┘
```

### 系统预设模板

| 模板名       | 基础风格    | 特点                           |
| ------------ | ----------- | ------------------------------ |
| 甄嬛传式宫斗 | web_gongdou | 一语双关、微表情描写、权力暗示 |
| 琅琊榜式权谋 | web_gongdou | 布局深远、智谋对决、配角立体   |
| 金庸经典武侠 | jin_yong    | 侠义气质、武功意境、江湖规矩   |

### 使用流程

1. **系统启动时**：`StyleTemplateService.initializeSystemTemplates()` 创建系统模板
2. **用户选择模板**：`setProjectTemplate(projectId, templateId)` 关联项目
3. **写作时**：`getMergedStyleConfig(projectId)` 获取合并后的完整配置
4. **生成提示词**：`fullPrompt` 包含详细的风格要求、示例、避免模式

## 实现复杂功能的验证流程

> 教训来源：三层风格配置实现中，首次 Dry Run 发现 3 个问题，深度 Dry Run 又发现 1 个严重问题

### 强制检查清单

**每写完一个方法后**：

```
□ 这个方法被谁调用？
□ 调用点传了所有必要参数吗？
□ 返回值被正确使用了吗？
```

**每写完一个服务后**：

```
□ 在 Module 中注册了吗（providers + exports）？
□ 需要初始化的方法在 onModuleInit 中调用了吗？
□ 依赖注入的服务在构造函数中声明了吗？
```

**涉及数据合并/转换时**：

```
□ 手动模拟 2-3 组数据，追踪合并结果
□ 嵌套对象是浅合并还是深合并？
□ 数组是覆盖还是追加？
□ undefined/null 如何处理？
```

**端到端验证**：

```
□ 从用户操作开始，追踪到最终输出
□ 每个边界情况（空值、缺失、删除）会怎样？
□ 类型断言 (as unknown as T) 的数据结构真的匹配吗？
```

### 常见陷阱

| 陷阱         | 表现                        | 防范                   |
| ------------ | --------------------------- | ---------------------- |
| 调用点遗漏   | 方法写好了但没人用          | 写完立即搜索调用点     |
| 浅合并覆盖   | `{...a, ...b}` 丢失嵌套数据 | 复杂对象必须深合并     |
| 初始化遗漏   | 服务方法存在但未执行        | 检查 onModuleInit      |
| 类型断言欺骗 | `as T` 绕过检查，运行时崩溃 | 添加运行时校验或默认值 |
| 重复查询     | 先查一次检查，再查一次使用  | 合并为单次查询         |

### 类型检查通过 ≠ 逻辑正确

```typescript
// 编译通过，但逻辑错误
const merged = { ...defaults, ...template }; // 浅合并丢数据

// 编译通过，但运行时可能崩溃
const rules = data as unknown as DialogueRules; // JSON 结构可能不匹配
rules.techniques.join(); // 如果 techniques 不存在就崩溃
```

**解决方案**：

1. 复杂合并用递归深合并
2. 类型断言前确保有默认值兜底
3. 访问嵌套属性前检查存在性或用可选链
