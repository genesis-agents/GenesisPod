# 报告模板规范 (Report Template Specification)

> 版本: 1.0 | 状态: 已实现 | 日期: 2026-03-07

本文档定义 Topic Insights 报告中 **每种内容类型** 的精确格式规则。
所有 Prompt 指令、后处理函数、前端渲染组件必须与本规范对齐。

---

## 全局规则

| 规则         | 说明                                                                                                                                           |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| 文风         | 第三人称为主（"数据显示..."），第一人称为辅（每子节最多 1 次"分析表明"）                                                                       |
| 术语         | 首次出现标注英文原文，如：能力密度（Capability Density）；后续统一中文                                                                         |
| 简体中文     | 全文简体，禁止繁体字                                                                                                                           |
| 公式         | KaTeX：行内 `$...$`，独立 `$$...$$`；一个表达式在同一对 $ 内，禁止拆分                                                                         |
| 分割线       | 正文禁止 `---`（仅参考文献前允许一条）                                                                                                         |
| 禁止泄露     | 字数统计、Agent/Leader 角色名、教材引用、内部标注                                                                                              |
| **文本对齐** | **所有编号/列表内容的续行，必须与编号后的正文对齐，不与编号本身对齐**。前端 CSS 通过 `list-style-position: outside` + 适当 `padding-left` 实现 |

**文本对齐示意（全局规则）：**

```
正确:                             错误:
1. 第一行内容文字对齐在这里，     1. 第一行内容文字对齐在这里，
   续行与"第"字对齐               续行与"1."对齐

- 第一行内容文字对齐在这里，     - 第一行内容文字对齐在这里，
  续行与"第"字对齐               续行与"-"对齐
```

## 类型 1: 报告标题

**Markdown 格式：**

```markdown
# 基础大模型技术发展趋势

> 生成时间：2026-03-07
```

| 项目     | 规则                       |
| -------- | -------------------------- |
| 来源     | 自动从 `topic.name` 生成   |
| 日期     | 引用块内，自动插入当前日期 |
| 前端渲染 | `<h1>` + 灰色日期行        |
| Prompt   | 无（框架自动生成）         |
| 后处理   | 无                         |

## 类型 2: 执行摘要

**Markdown 格式：**

```markdown
## 执行摘要

**核心论断：基础大模型正从规模竞赛转向效率竞赛。**

研究范围覆盖 2024-2026 年全球基础模型技术、市场与政策格局。

### 核心发现

1. **开源模型已逼近闭源 90% 性能。** 数据显示 Llama 3.1 在 MMLU 上达到 87%[5][12]。
2. **MoE 架构成为主流路线。** DeepSeek V3 的 top-k 稀疏门控将推理成本降低 60%[23][45]。
3. **缩放定律从参数转向数据均衡。** Chinchilla 最优配置降低训练成本 30%[1][7]。

### 关键指标

| 指标             | 数值 | 来源 |
| ---------------- | ---- | ---- |
| 开源/闭源性能比  | 90%  | [5]  |
| MoE 推理成本降幅 | 60%  | [23] |

### 风险预警

1. 数据合规与隐私要求将推高合成数据投入
2. AGI 路线不确定性加大政策摇摆风险

### 行动建议

1. **企业决策者：** 优先部署 MoE 架构降低推理成本
2. **投资者：** 关注开源生态的商业化拐点
3. **技术从业者：** 聚焦高效架构与系统级优化
```

| 项目        | 规则                                                                                                                                                                                           |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 结构        | McKinsey SCR 框架，6 个区块各有 `###` 子标题：核心论断(加粗段落) → 背景(段落) → `### 核心发现`(编号列表) → `### 关键指标`(表格) → `### 风险预警`(编号列表) → `### 行动建议`(编号列表+角色加粗) |
| 子标题      | 每个区块必须有 `###` 标题区分，禁止多段编号列表无标题并列                                                                                                                                      |
| 字数        | 400-600 字                                                                                                                                                                                     |
| 加粗        | 核心论断整句加粗；核心发现每条第一句（判断）加粗，第二句（数据）不加粗；行动建议角色名加粗                                                                                                     |
| 表格位置    | 表格与正文对齐（不缩进在编号列表下），位于 `### 关键指标` 标题之后                                                                                                                             |
| 引用块      | 禁止                                                                                                                                                                                           |
| Prompt 常量 | `EXECUTIVE_SUMMARY_FORMAT`                                                                                                                                                                     |
| 后处理      | `enforceExecSummarySections`                                                                                                                                                                   |
| 前端渲染    | 标准 markdown                                                                                                                                                                                  |

## 类型 4: 章节要点 (Chapter Highlights)

**Markdown 格式：**

```markdown
> **本章要点**
>
> - Transformer 变体架构主导 2026 年基础模型生态
> - MoE 稀疏激活将推理成本降低 50%-70%
> - 缩放定律从参数规模转向数据与算力均衡
```

**前端渲染效果：**

```
 ┃  本章要点
 ┃
 ┃  ● Transformer 变体架构主导 2026 年基础模型生态
 ┃  ● MoE 稀疏激活将推理成本降低 50%-70%
 ┃  ● 缩放定律从参数规模转向数据与算力均衡
 ┃
    ↑ 蓝色竖线 (border-left)      ↑ 浅蓝背景 (bg-blue-50)
```

| 项目         | 规则                                                                                                                                                 |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| 位置         | 每个 `## N. 维度名` 开头，第一个 `###` 之前                                                                                                          |
| 标题行       | `> **本章要点**`（英文：`> **Chapter Highlights**`）                                                                                                 |
| 要点行       | 每行必须以 `> - ` 开头（禁止省略 `- `）                                                                                                              |
| 条数         | 3-5 条                                                                                                                                               |
| 每条字数     | 严格 ≤30 字（含标点）                                                                                                                                |
| 每条内容     | 完整结论性判断句，必须是独立可读的完整句子                                                                                                           |
| 禁止内容     | 数学公式、技术细节、引用标注 [N]、半截结尾                                                                                                           |
| Fallback     | LLM 未生成时，后端从 `keyFindings` 自动生成（截取前 60 字 + `> - ` 格式）                                                                            |
| Prompt 常量  | `CHAPTER_HIGHLIGHTS`                                                                                                                                 |
| 后处理       | `bulletifyBlockquoteItems`（确保所有行有 `- `）                                                                                                      |
| **前端渲染** | **样式化卡片**：蓝色左竖线 `border-l-4 border-blue-400` + 浅蓝背景 `bg-blue-50/60` + 圆角 `rounded-xl` + 圆点标记（复用章节视图 Key Takeaways 组件） |
| 前端识别     | `createMarkdownComponents` 的 `blockquote` 渲染器检测首行包含"本章要点"或"Chapter Highlights"时切换为卡片组件                                        |

## 类型 5: 维度正文段落

**Markdown 格式：**

```markdown
### 1.1. Transformer 基础原理

Transformer 架构于 2017 年提出，以编码器-解码器结构取代 RNN/LSTM 序列模型[73][74]。
**其核心创新在于自注意力机制实现了并行全局交互**，训练速度较 RNN 提升 3-5 倍[5][11]。

**技术栈收敛趋势**：2026 年变体演进聚焦稀疏（如 DeepSeek V3.2 top-k 掩码，O(N log N)）
与线性注意力（如 Qwen3-Next Gated DeltaNet，O(N)），推理延迟降 30%-50%[10][13]。
```

### 5a. 标题规则

| 项目        | 规则                                                                                                |
| ----------- | --------------------------------------------------------------------------------------------------- |
| 允许层级    | `###`（子节）和 `####`（子子节）                                                                    |
| 禁止层级    | `#`、`##`（框架控制）、`#####` 及更深                                                               |
| 编号        | 自动编号：`### N.M. 标题`、`#### N.M.K. 标题`                                                       |
| 子节数量    | 每维度最多 6-8 个 `###`                                                                             |
| Prompt 常量 | `HEADING_HIERARCHY`                                                                                 |
| 后处理      | `sanitizeHeadingLevels` + `numberSubHeadings` + `deduplicateHeadings` + `collapseExcessSubHeadings` |

### 5b. 段落规则

| 项目         | 规则                                                           |
| ------------ | -------------------------------------------------------------- |
| 段落长度     | ≤400 字，超长自动在句子边界拆分                                |
| 冒号前总结语 | 段落开头 3-25 字短语 + `：` → 自动加粗为 `**短语**：`          |
| 引用密度     | 每千字 10-15 处，均匀分布                                      |
| 引用格式     | `[N]` 渲染为 `<a href="#ref-N">[N]</a>` 可点击跳转             |
| 后处理       | `splitWallOfText` + `boldSummaryPrefixes` + `linkifyCitations` |

### 5c. 加粗规则

| 项目        | 规则                                                |
| ----------- | --------------------------------------------------- |
| 限额        | 每子节（`###` 下）最多 2 处加粗                     |
| 加粗对象    | 仅核心判断性语句（如"开源模型已逼近闭源 90% 性能"） |
| 加粗长度    | 单处加粗 ≤30 字                                     |
| 禁止加粗    | 单独的数字/百分比/倍数（如 ~~**68倍**~~）           |
| 扫描层      | 读者仅看加粗内容即可获取核心论点                    |
| Prompt 常量 | `FORMATTING_LIMITS` 加粗节                          |
| 后处理      | `limitBoldFormatting`（全文 >120 处时自动削减）     |

### 5d. 枚举规则

| 项目     | 规则                                                                                                                                                                                                |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 触发模式 | 段落内出现 `一是...二是...` / `一方面...另一方面...` / `首先...其次...` / `其一...其二...` / `第一...第二...`                                                                                       |
| 处理方式 | **拆分为无序列表**：枚举前的引导句保留为段落，每个枚举项变为 `- ` 列表项                                                                                                                            |
| 示例输入 | `在技术栈层面，可观察到三条路线：一是以通用语言模型为核心...二是以物理AI为代表...三是结合行业数据...`                                                                                               |
| 示例输出 | `在技术栈层面，可观察到三条路线：\n\n- 以通用语言模型为核心，通过规模化与工具调用增强能力[2]\n- 以物理AI路线为代表，聚焦三维空间与动力学预测[127]\n- 结合行业数据与业务流程的行业基础模型路线[125]` |
| 后处理   | `splitEnumerationToList`（新函数，替代 `boldEnumerationMarkers`）                                                                                                                                   |

## 类型 6: 列表

**Markdown 格式：**

**Markdown 格式（源码层级一致写 `-`，由前端 CSS 区分渲染）：**

```markdown
1. **开源模型已逼近闭源 90% 性能。** Llama 3.1 在 MMLU 上达到 87%[5]
2. **MoE 架构成为主流。** 推理成本降低 60%[23]
   a. DeepSeek V3 的 top-k 稀疏门控方案
   b. Mixtral 的 Expert-Choice 路由策略

- 隐私合规要求增加数据合成投入
- 模型评估标准从通用转向领域特定
  - 医疗领域采用 MedQA 基准
  - 法律领域采用 LegalBench 基准
```

**层级标识符规则：**

| 列表类型 | 一级标识              | 二级标识            | 前端 CSS                       |
| -------- | --------------------- | ------------------- | ------------------------------ |
| 有序列表 | `1. 2. 3.` 阿拉伯数字 | `a. b. c.` 小写字母 | `list-style-type: lower-alpha` |
| 无序列表 | `●` 实心圆            | `○` 空心圆          | 一级 `disc`，二级 `circle`     |

**渲染效果：**

```
  1. 开源模型已逼近闭源 90% 性能
  2. MoE 架构成为主流
     a. DeepSeek V3 的 top-k 稀疏门控方案
     b. Mixtral 的 Expert-Choice 路由策略

  ● 隐私合规要求增加数据合成投入
  ● 模型评估标准从通用转向领域特定
    ○ 医疗领域采用 MedQA 基准
    ○ 法律领域采用 LegalBench 基准
```

| 项目         | 规则                                                                                   |
| ------------ | -------------------------------------------------------------------------------------- |
| 有序列表一级 | `1. 2. 3.`（阿拉伯数字），禁止中文数字"一、二、三"                                     |
| 有序列表二级 | `a. b. c.`（小写字母），前端 `list-style-type: lower-alpha`                            |
| 无序列表一级 | Markdown 写 `- `，前端渲染为 `●`（实心圆），`list-style-type: disc`                    |
| 无序列表二级 | Markdown 写 ` -`（缩进），前端渲染为 `○`（空心圆），`list-style-type: circle`          |
| 嵌套         | 最多 2 层                                                                              |
| 每项字数     | ≤100 字，超长应拆分为多条或改用段落                                                    |
| 禁止         | 结论性段落放在列表末尾作为最后一个列表项；不同层级使用相同标识符                       |
| Prompt 常量  | `FORMATTING_LIMITS` 列表节                                                             |
| 后处理       | `repairOrderedListContinuity` + `truncateLongListItems` + `separateTrappedConclusions` |
| 前端实现     | `createMarkdownComponents` 中 `ul`/`ol` 组件根据嵌套深度设置不同 `list-style-type`     |

## 类型 7: 正文引用块 (Blockquote)

**用途：** 在维度正文的 `###` 子节中高亮一条核心判断，与章节要点（类型 4）互补——章节要点是"全章概览"，引用块是"子节内最重要的一句话"。

**Markdown 格式：**

```markdown
### 1.3. 开源与闭源竞争格局

段落段落段落...

> **开源路线正在从"跟随"转向"引领"，2026 年将形成与闭源的结构性竞争。**

段落段落段落...
```

| 项目        | 规则                                                                           |
| ----------- | ------------------------------------------------------------------------------ |
| 全文限额    | 最多 8 个（含所有维度，不含"本章要点"）                                        |
| 每维度限额  | **最多 1 个**（选该维度最核心的单条判断）                                      |
| 每个字数    | ≤80 字                                                                         |
| 内容要求    | 必须是独立判断句，不得与本章要点中的任何一条重复                               |
| 格式        | `> **判断句内容。**`（整句加粗）                                               |
| 禁止        | 每段结尾加引用块总结；补充节（跨维度/风险/战略）使用引用块；引用块内放多条内容 |
| Prompt 常量 | `FORMATTING_LIMITS` 引用块节                                                   |
| 后处理      | 无额外处理                                                                     |
| 前端渲染    | 默认 prose blockquote 样式（灰色左边框 + 灰色背景），与类型 4 蓝色卡片视觉区分 |

## 类型 8: 图表 (Figures)

**Markdown 格式（正文中）：**

```markdown
分析表明，Transformer 变体在 2024-2026 年持续演进，稀疏化与线性化成为两大方向。
如图 2 所示，MoE 架构的 FLOPs 效率显著优于稠密模型[77][94]。

<!-- chart:d0-fig-transformer-evolution -->

此路径放大预训练收益，推动长上下文范式转型[79]。
```

### 8a. 正文引用规则

| 项目     | 规则                                                                                          |
| -------- | --------------------------------------------------------------------------------------------- |
| 占位符   | `<!-- chart:dN-id -->` 是唯一的图表嵌入方式                                                   |
| 编号引用 | **允许**自然语言引用如"如图 2 所示"、"图 3 展示了..."，编号须与 FigureRenderer 卡片内编号一致 |
| 禁止     | "图N. 标题"独立行（标题由卡片渲染）；"来源: 证据[N]"独立行（来源由卡片渲染）；"图N:M..."乱码  |

### 8b. FigureRenderer 卡片渲染规则

```
┌─────────────────────────────────────────────────┐
│ 图 2. 开源模型参数规模在 2024-2026 年稳定在      │  ← 编号(蓝色) + 标题(黑色)
│       120B-235B 区间                              │
├─────────────────────────────────────────────────┤
│                                                   │
│              [图片/图表内容区域]                    │  ← 引用图: <img>; 生成图: Recharts
│                                                   │
├─────────────────────────────────────────────────┤
│ 说明文字（灰色小字）                               │  ← chart.description
│ 来源: [23]                                        │  ← 蓝色 badge，可点击跳转
└─────────────────────────────────────────────────┘
```

| 项目        | 规则                                                                       |
| ----------- | -------------------------------------------------------------------------- |
| 编号        | `图 N.`（蓝色加粗），全文顺序编号                                          |
| 标题        | 完整的发现性句子（McKinsey 标准），禁止"示意"/"概念"等弱化词               |
| 说明        | `chart.description`，灰色小字，位于图表下方                                |
| 来源        | `来源: [N]` badge，蓝色背景，可点击跳转到参考文献                          |
| 图片失败    | 占位符：图标 + "图片加载失败" + 标题（灰色小字）                           |
| 每维度      | 最多 2 个图表                                                              |
| 全文        | 最多 12-14 个                                                              |
| 位置        | 紧跟首次提及段落之后，禁止集中在章节末尾                                   |
| Prompt 常量 | `CHART_STANDARDS`                                                          |
| 后处理      | `resolveChartPlaceholders` + `stripInternalFigureNotation`（清理冗余文字） |
| 前端组件    | `<FigureRenderer>` 统一渲染所有图表类型                                    |

## 类型 9: 表格

**Markdown 格式：**

```markdown
| 指标                  | 2024 | 2025 | 2026 | 来源 |
| --------------------- | ---- | ---- | ---- | ---- |
| 开源/闭源性能比       | 75%  | 85%  | 90%  | [5]  |
| 推理成本 ($/M tokens) | 15.0 | 3.0  | 0.6  | [23] |
```

| 项目     | 规则                                                                          |
| -------- | ----------------------------------------------------------------------------- | --- | ------------------------------ |
| 格式     | 标准 GFM markdown 表格                                                        |
| 分隔行   | `                                                                             | --- | ` 必须存在，列数必须与表头匹配 |
| 脚注行   | 最后一行第一格 >50 字且其余格为空 → 提取为表格下方独立段落                    |
| 前端渲染 | 标准 `<table>`（prose 样式）                                                  |
| 后处理   | `repairMarkdownTables`（修复缺失分隔行/列数不匹配） + `extractTableFootnotes` |

## 类型 10: 补充章节 (跨维度关联分析 / 风险评估 / 战略建议)

**Markdown 格式：**

```markdown
## 跨维度关联分析

### 因果链分析

核心架构优化如稀疏注意力导致关键技术创新加速 MoE 发展，
**这进一步推动前沿模型水平跃升至万亿参数**，时间窗口 6-12 个月[125][82]。

### 系统性风险

- 资本与算力集中在少数主体，单点失败可能产生全球放大效应
- 对齐安全与政策伦理双重约束下的合规成本上升
```

| 项目        | 规则                                                                               |
| ----------- | ---------------------------------------------------------------------------------- |
| 标题        | `## 节名`（框架控制）+ `### 子标题`（AI 生成，必须有子标题组织结构）               |
| 加粗        | 每节最多 2 处核心判断句                                                            |
| 引用块      | **禁止**（引用块预算留给维度章节）                                                 |
| 列表项      | ≤100 字，结论性内容用段落而非列表项                                                |
| 禁止        | 用加粗文本代替标题（如 ~~**技术架构：**~~ → 应为 `### 技术架构`）                  |
| 子标题示例  | 跨维度：`### 因果链分析`、`### 维度对比`；战略建议：`### 企业决策者`、`### 投资者` |
| Prompt 常量 | `SYNTHESIS_FORMATTING`                                                             |
| 后处理      | 与全文共享 `postProcessFinalReport` 管线                                           |

## 类型 12: 参考文献

**Markdown 格式：**

```markdown
---

# 参考文献

[1] [Attention Is All You Need](https://arxiv.org/abs/1706.03762). 访问日期: 2026-03-01

[2] [DeepSeek V3 Technical Report](https://deepseek.com/research/v3). 访问日期: 2026-02-15

[3] [Stanford HAI AI Index Report 2026](https://aiindex.stanford.edu). 访问日期: 2026-03-05
```

**渲染效果（用户看到的）：**

```
  ───────────────────────────────────

  参考文献

  [1] Attention Is All You Need. 访问日期: 2026-03-01
      ↑ 蓝色超链接，点击跳转到 arxiv.org

  [2] DeepSeek V3 Technical Report. 访问日期: 2026-02-15
      ↑ 蓝色超链接，点击跳转到 deepseek.com

  [3] Stanford HAI AI Index Report 2026. 访问日期: 2026-03-05
      ↑ 蓝色超链接，点击跳转到 aiindex.stanford.edu
```

**核心原则：标题即超链接，URL 不可见。** 用户只看到标题文字（蓝色可点击），不看到任何原始 URL。

| 项目     | 规则                                                                                                                   |
| -------- | ---------------------------------------------------------------------------------------------------------------------- |
| 格式     | `[N] [标题](URL). 访问日期: YYYY-MM-DD`                                                                                |
| 渲染     | **标题显示为蓝色超链接**，原始 URL 完全不可见，点击标题跳转到原文                                                      |
| 编号     | `[N]` 为纯文本编号，同时带有锚点 `<a id="ref-N">` 供正文引用跳转                                                       |
| 正文引用 | 正文中 `[N]` → `<a href="#ref-N">[N]</a>` 蓝色可点击，跳转到对应参考文献                                               |
| URL 清理 | `http://` → `https://` 自动升级                                                                                        |
| 去重     | 同一 URL 保留首次出现，合并引用编号                                                                                    |
| 垃圾过滤 | 过滤 google.com/search、favicon.ico、localhost 等无效链接                                                              |
| 后处理   | `filterJunkReferences` + `upgradeHttpToHttps` + `deduplicateReferencesByUrl` + `anchorReferences` + `linkifyCitations` |

## 类型 13: 数学公式

**Markdown 格式：**

```markdown
行内: 缩放定律表述损失 $L \propto N^{-\alpha} + D^{-\beta}$，其中 $N$ 为参数量。

独立:

$$
L(N,C,D) \approx \frac{a}{N^{\alpha}} + \frac{b}{C^{\beta}} + \frac{c}{D^{\gamma}}
$$
```

| 项目       | 规则                                                              |
| ---------- | ----------------------------------------------------------------- |
| 行内       | `$...$`，一个表达式必须在同一对 $ 内                              |
| 独立       | `$$...$$`，前后各一个空行                                         |
| 简单表达式 | 可用 Unicode：O(n²)、√n                                           |
| 禁止       | 用代码块展示公式；拆分写法如 `$A$ $\in$ $B$`（正确：`$A \in B$`） |
| 前端渲染   | KaTeX（`rehype-katex`）                                           |
| 后处理     | `mergeAdjacentMathBlocks`（合并相邻碎片化 $...$ 块）              |
