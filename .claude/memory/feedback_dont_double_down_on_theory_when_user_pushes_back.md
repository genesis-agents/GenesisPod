诊断时抓到一个"看起来对"的信号（如日志里的 error 行）后，不要锁死结论一路辩护；用户带证据反驳时，**从头重查**，把用户最早说的症状当第一线索。

**Why（2026-05-21 AI Ask 房间事故）**：用户一开始就说"代理电脑上看不到输出"。我却抓住后端日志的 `xAI 402` + pricing-registry 警告，连续三轮断言"没额度→key 不对→max_tokens=16000 预扣"，每次被打脸就换个理由继续维护"402 是根因"。用户连发：有额度、同一个 key、用 BYOK、**其实过会儿回复了只是代理看不到**，最后骂"一本正经地胡说八道"。真因正是他第一句话——代理实时送达，402 只是支线。我重复了 [[feedback-byok-must-check-layers-above-chat]] 里写过的元教训："写了规则≠按规则做"。

**How to apply**：

- 用户反驳一次 = 信号我可能错；反驳两次 = 必须推倒重来，别再打补丁。
- 优先采信用户的**第一手现象描述**（他在现场、我看不到屏幕），代码/日志只是佐证，不能反压现象。
- 区分"主因 vs 支线"：日志里同时有多个 error 时，先问"哪个真的导致用户看到的症状"，别抓第一个显眼的就开干（呼应 CLAUDE.md「分析先行/端到端追踪」）。
- 截图/日志只证明它**证明的那一点**，别外推（402 只证明那一次 xAI 调用失败，不证明"没输出"）。
