# "沉淀"是默认 task，不是事后补丁

**Why**：2026-04-29 端到端验证暴露 5 个 UI bug 后，用户提醒"还有你最后的沉淀呢"。我才想起写 memory，事后补 task #13。这违反了"做一次必须沉淀一次"的工作纪律。

如果不形成默认任务，每次都依赖外部提醒 → 教训写不下来 → 同样的错误下次还会犯。

**How to apply**：

1. **每次创建 task 列表时**（TaskCreate 多个 task 规划工作时），**最后一项必须是「沉淀本轮经验到 memory」**，不要等收尾。

   例：

   ```
   #1 PR-1a 后端 endpoint
   #2 PR-2 前端 UI
   #3 PR-3 报告版式
   #4 类型检查 + 端到端验证 + 推送
   #5 沉淀经验到 memory  ← 默认存在
   ```

2. **沉淀的判断标准**：本轮工作中出现以下任一即必须沉淀：
   - 用户的明确反馈 / 纠正（"不要这样"、"应该那样"）
   - 用户在我做错后提醒的修正方式
   - 我自己发现的根因 bug + 修复模式
   - 验证流程的疏漏（如本次"只查 DB 没看 UI"）

3. **沉淀时机**：在 push 最终 commit 之前，不是之后。否则 push 完用户接手验收，沉淀就被遗忘。

4. **沉淀范围**：每条 memory 必须含 Why（来自哪次教训）+ How to apply（下次怎么应用）。仅记录"做了什么"是流水账，不是沉淀。

5. **沉淀也是产出**：跟代码 commit 一样重要。不沉淀 = 工作只完成 80%。
