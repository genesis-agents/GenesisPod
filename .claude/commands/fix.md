# Bug Fix

按照项目规范修复 Bug。强制完整链路追踪，禁止跳步。

**问题描述**: $ARGUMENTS

## Phase 1: 定位（不写任何代码）

### 1.1 问题定位五问（修复前必答）

- [ ] 用户在哪个页面/路由？
- [ ] 用户执行了什么操作？
- [ ] 用户期望看到什么？
- [ ] 用户实际看到什么？
- [ ] 这个 UI 元素/功能在代码中的确切位置？

### 1.2 端到端路径追踪

从 UI 开始，追踪完整调用链，**每一步都必须 Read 源码确认**：

```
[UI组件位置] → [事件处理函数] → [Store方法/API调用]
→ [后端Controller] → [后端Service] → [数据库/外部调用] → [返回数据] → [前端渲染]
```

### 1.3 多位置检查

同一功能可能在多个位置渲染，用 Grep 搜索所有相关位置：

```bash
grep -r "关键函数名/组件名" --include="*.tsx" --include="*.ts" backend/src frontend/
```

### 1.4 输出诊断报告

格式：

```
Root Cause: [文件:行号] - [一句话描述]
影响范围: [列出所有需要修改的文件]
修复方案: [具体做什么]
```

**等待用户确认后再继续 Phase 2。如果用户说"直接修"或问题很明确，可以跳过等待。**

---

## Phase 2: 修复

1. 只修改 Phase 1 中确认的文件，不碰其他代码
2. 修改后立即跑对应的测试文件
3. 如果没有对应测试，写一个复现 bug 的测试用例

---

## Phase 3: 验证

1. 跑对应模块的测试：`npx -w backend jest --testPathPattern="模块名" --bail`
2. 跑类型检查：`npm run type-check`
3. 用 `git diff` 列出所有修改的文件，逐个确认变更在任务范围内

**只有测试通过 + 类型检查通过才能说 "done"。**

---

## 禁止行为

- ❌ 搜索关键词，修改第一个找到的
- ❌ 不读代码就猜 root cause
- ❌ 只修前端或只修后端
- ❌ 修改后不跑测试就说"修好了"
- ❌ 修改任务范围外的文件
- ❌ 只改一个位置（不检查是否有重复组件）
