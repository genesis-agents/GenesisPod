# AI 大语言模型评估方法体系

随着大语言模型能力边界的不断扩展，如何系统化、可重复地度量模型表现成为工程实践中的核心问题。一个完备的评估方案需要同时覆盖知识广度、推理深度、生成质量与对齐安全四个维度，并辅以可量化的指标和稳定的基准数据集。单纯依赖某一类指标极容易得到误导性的结论：仅看选择题准确率会忽略生成质量与开放对话表现，仅看自动指标会错过语义改写带来的真实改进，仅看人类偏好则可能放大评委本身的偏见。本文梳理业界主流评估方法、关键指标及代表性基准，并给出工程落地的组合策略，方便团队在迭代过程中快速判断每个版本相对基线的位移方向。

## 知识与综合能力基准

### MMLU

MMLU 全称 Massive Multitask Language Understanding，由 Hendrycks 等人于 2020 年提出，覆盖 STEM、人文、社会科学、其他四大类共五十七个学科，包含约 14000 道四选一选择题。MMLU 用于衡量模型在零样本或少样本设置下的跨学科知识水平，其结果通常作为通用模型能力的第一参考指标。GPT-4 在 MMLU 上的五样本准确率约为 86%，超越了多数早期开源模型，但近年来如 Llama 3、Qwen 2、DeepSeek V2 等开源模型已可逼近这一水平。MMLU 的弱点是题目泄漏风险高，部分测试集已被各家训练语料覆盖，因此社区推出了 MMLU-Pro、MMLU-Redux 等更新版本以重建难度梯度，并将选项从四个扩到十个以降低猜测命中率。需要注意 MMLU 仅测知识与浅层推理，不能反映长链条规划或工具调用能力，应与其他基准互补。

### BIG-Bench Hard

BIG-Bench 是 Google 于 2022 年发布的超大规模任务集合，原始版本包含两百余个任务，覆盖逻辑、社会偏见、代码、翻译、数学推理等子域。BIG-Bench Hard 简称 BBH，从中筛选出 23 个挑战性最强、模型一致表现较差的任务，专门用于压力测试。该基准强调思维链推理的必要性，模型在加入 chain-of-thought 提示后通常有显著提升，这种"加思维链就涨分"的现象本身被视为模型具备推理能力的间接证据。BBH 也被广泛用于 SFT 后回归测试，以判断指令微调是否削弱了基础推理能力。在 BBH 评估时务必固定 few-shot 例子顺序与思维链模板，否则结果方差极大。

## 代码与编程能力基准

### HumanEval

HumanEval 由 OpenAI 于 2021 年提出，包含 164 道手写的 Python 编程题目，每题给出函数签名、文档字符串和若干隐藏单元测试。评估指标为 pass@k，即模型采样 k 次中至少有一次通过所有测试用例的概率。pass@1 通常被作为最严苛的实用化指标，pass@10 与 pass@100 则用于衡量模型分布的覆盖能力。HumanEval 是衡量模型代码生成正确性最广泛采用的微型基准，但题目规模偏小、覆盖范围有限，社区因此推出 MBPP、APPS、SWE-bench、CodeContests、BigCodeBench 等更难、更贴近真实工程的扩展。Claude 3.5 Sonnet 在 HumanEval 上 pass@1 已突破 92%，但 SWE-bench 这类需要在真实仓库中跨多文件改动的任务仍然挑战性十足，强模型解决率也仅约 50%。HumanEval+ 通过引入更严格的隐藏测试集纠正了原版用例覆盖不足的问题，更适合现代评估。

### LiveCodeBench

为应对训练集泄漏问题，LiveCodeBench 通过持续采集近期 Codeforces、LeetCode、AtCoder 比赛题构造动态测试集，强调时间窗口内未见过的新题。该基准要求每隔一段时间公布题目快照，并按训练截止日期切分模型可见与不可见区间，更能反映模型真实泛化能力而非记忆。其报告通常给出多个时间窗口下的 pass@1 曲线，研究者可观察模型在新题上的性能衰减。LiveCodeBench 还细分了"代码生成"、"自我修复"、"测试用例生成"等子任务，便于诊断模型在代码全流程中的具体弱点。

## 生成质量自动指标

### BLEU 与 ROUGE

BLEU 起源于机器翻译评估，通过 n-gram 重合度衡量候选与参考之间的相似性，公式中加入 brevity penalty 惩罚过短输出。ROUGE 系列则更常用于摘要任务，包括 ROUGE-N、ROUGE-L、ROUGE-W 等变体，关注召回率而非精确率。这两类指标实现简单、可重复，但都对语义改写不敏感，模型用同义改写替换原词时分数会大幅下降，因此现代评估通常将 BLEU 和 ROUGE 与 LLM-as-Judge 类语义指标组合使用。对于结构化输出如 SQL 或 JSON，BLEU 几乎完全失效，应改用执行准确率或字段级精确率。即便如此，BLEU 与 ROUGE 在做长期趋势监测时仍有不可替代的价值，因为它们完全确定性、不会随评委模型更新而漂移。

### 困惑度

困惑度是语言建模阶段的核心指标，定义为序列条件概率几何平均的倒数，直观上反映模型对真实文本分布的拟合程度。困惑度越低代表模型对测试集预测越准。困惑度在预训练评估中是必备指标，但对指令微调后的对话模型已不能直接刻画用户体验，因为模型可能拟合了人类偏好但牺牲了原始分布的预测精度。在 RLHF 阶段，研究者通常会同时跟踪困惑度漂移与人类偏好得分，以检测是否过拟合到奖励模型。困惑度对 tokenizer 极其敏感，跨 tokenizer 比较毫无意义，报告时必须声明 tokenizer 版本与测试集来源。

## 模型作为评估者

### LLM-as-Judge

LLM-as-Judge 是近年兴起的范式，让一个能力更强的模型（如 GPT-4 或 Claude Opus）扮演评委角色，对候选输出按指定 rubric 给出胜负或评分。该方法解决了 BLEU 与 ROUGE 对语义改写不敏感的痛点，且能覆盖创意写作、对话连贯性、安全合规等难以量化的维度。代表性框架包括 MT-Bench、AlpacaEval、Arena-Hard 与 Chatbot Arena 的离线复刻版本。LLM-as-Judge 的主要风险是评委模型自身偏见，例如位置偏见、长度偏见和自我偏好偏见，社区通常通过双盲交换位置、控制长度上限、屏蔽模型名等手段缓解。此外，最新研究表明使用同一家族的模型互评会引入显著的家族偏好，建议采用异家族评委或者多评委集成投票，以提高结果稳健性。

### Chatbot Arena

Chatbot Arena 由 LMSYS 团队维护，采用两两对战的众包模式，由真实用户对匿名模型输出投票，最终以 Elo 评分对所有参赛模型进行排名。该评估方式贴近真实使用场景，结果较难被刷分操纵，已成为社区公认的综合能力风向标之一。其分类榜单还细分了编程、长上下文、多语种、视觉、推理等子赛道，便于研究者按需查阅。但 Chatbot Arena 也受到样式偏好的影响，用户倾向于偏好排版精美、措辞礼貌的答案，因此 Arena 排名不能完全等同于"实用价值"。

## 对齐与安全评估

对齐评估关注模型在拒答、毒性、偏见、隐私、越狱抵抗等维度的表现。主流基准包括 HarmBench、TruthfulQA、BBQ、AdvBench，以及红队自动化框架如 PAIR、TAP、AutoDAN。这类基准的输出通常不仅看是否通过，还会记录失败模式以指导后续 RLHF 数据采集。安全评估须与能力评估解耦报告，避免出现"安全分高但能力下降"的回归被均值平均掉。生产环境部署前还应跑专有领域的红队脚本，覆盖具体业务可能遭遇的滥用模式。

## 工具调用与 Agent 能力评估

随着模型从单轮对话向多步工具调用演进，专门的 Agent 评估也开始兴起。主流基准包括 ToolBench、API-Bank、AgentBench 与 GAIA。ToolBench 侧重单一工具调用的格式正确性与参数填充，API-Bank 覆盖多 API 编排，AgentBench 在多个交互式环境中评估规划、自我修复、长程记忆等能力，GAIA 则以真实世界问题（需要联网、读取附件、跨步推理）作为终极挑战。Agent 评估的指标已超越 pass@k，常用维度还包括成功率、平均步数、工具调用错误率与代价（token 与外部 API 费用）。一个被忽略的细节是：Agent 任务的方差极大，单次抽样足以让排名颠倒，建议至少跑 5 次取均值并报告标准差。

## 长上下文与多模态评估

随着上下文窗口扩展到百万级别，长上下文评估成为独立赛道。常用基准包括 Needle in a Haystack、LongBench、RULER 与 BABILong，分别考察单点检索、多文档问答、合成压力测试与极端长序列下的逻辑链。对于多模态模型，业界使用 MMMU、MathVista、ChartQA、DocVQA 等基准覆盖跨模态推理、图表理解、文档抽取与视觉数学题。评估时应注意上下文位置敏感性，关键信息放在不同位置（开头、中间、结尾）模型表现差异巨大，这种"中间迷失"现象是当前长上下文模型的共性缺陷。

## 评估实践建议

实际工程中应避免单一指标决策。一般组合是：用 MMLU 与 BBH 衡量综合能力上限，用 HumanEval 与 LiveCodeBench 衡量代码能力，用 BLEU 与 ROUGE 衡量结构化生成，用 LLM-as-Judge 衡量开放生成质量，用 HarmBench 与 TruthfulQA 衡量安全，用 RULER 与 LongBench 衡量长上下文。所有基准须固定提示模板、采样参数、随机种子，并报告版本号以便复现。评估应贯穿训练全流程，从预训练困惑度到 SFT、RLHF 各阶段对比，以及部署前的回归基线。建议建立内部黄金集，与公开基准并行跟踪，以发现公开基准未能覆盖的业务专属问题。内部黄金集应来自真实生产对话脱敏抽样，按任务类型分层覆盖，且需要定期刷新以避免被自家训练流程吸收。最后，所有评估结果都应回到产品价值视角解读，单点分数提升不等于用户体验提升，须辅以小流量灰度验证、用户留存与活跃指标，以及主观满意度问卷构成的多源证据链来决策模型是否上线。值得一提的是，评估基础设施本身也需要持续投入：稳定的运行环境、可复现的种子管理、足够的并发吞吐、清晰的版本号留存与可追溯的结果归档，缺一不可。许多团队在快速迭代阶段因为评估流水线不稳，反复怀疑训练改动是否真的有效，反而拖慢迭代节奏。一条经验法则是：把评估流水线当作和训练流水线同等重要的资产投入资源建设，长期收益远大于短期省下的工程时间，因为评估基础设施一旦稳定下来便会持续受益于团队所有后续的模型实验。
