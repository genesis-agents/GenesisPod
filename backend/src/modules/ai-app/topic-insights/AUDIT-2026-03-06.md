## Executive Summary

Topic Insights is the platform's最大且最复杂的 AI App 模块，实现了 **Leader-Agent 多步骤研究编排** 系统。整体架构设计合理、分层清晰，但随着功能迭代积累了显著的 **God Service** 和 **配置散落** 问题。

## 1. Architecture & Dependency Injection (85/100)

### Strengths

- **78 providers** well-organized across 7 functional domains (core, dimension, report, data, collaboration, quality, monitoring)
- **100% Facade compliance** — zero direct `ai-engine/` internal imports found
- **God Service decomposition** already started: `ResearchMissionService` delegates to 3 sub-services (`MissionObservabilityService`, `MissionKernelBridgeService`, `MissionNotificationService`)

## 2. Research Execution Flow (75/100)

### Mission State Machine

```
PLANNING → EXECUTING → COMPLETED
    ↓          ↓
  FAILED    FAILED / CANCELLED
```

**Issue**: `PLAN_READY` and `REVIEWING` states are declared in Prisma enum but **never used** in any code path. This suggests an incomplete approval workflow.

### Execution Architecture

```
User Request
  → MissionLifecycleService.createMission() [returns immediately, 10min planning timeout]
    → LeaderService.planResearch() [LLM-based planning]
      → createTasksFromPlan() [dimension_research × N + quality_review + report_synthesis]
        → MissionExecutionService.executeDynamicScheduler()
          → Promise.race() with MAX_CONCURRENCY=8, deadlock detection at 60s
            → Per-task: DimensionMissionService.executeDimensionMission()
              → Phase 1: Data collection (multi-source parallel)
              → Phase 2: Section writing (multi-section parallel)
              → Phase 3: Leader review (multi-round)
```

### Key Findings

| Aspect                            | Status      | Detail                                                          |
| --------------------------------- | ----------- | --------------------------------------------------------------- |
| **Non-blocking mission creation** | Good        | Returns immediately, async planning with 10min timeout          |
| **Dynamic concurrency**           | Good        | 4-8 slots based on provider count, avoids DB pool exhaustion    |
| **Failure isolation**             | Good        | One dimension failure doesn't block others (try-catch per task) |
| **Deadlock detection**            | Good        | 30 consecutive waits x 2s = 60s timeout                         |
| **Per-task timeout**              | **Missing** | Relies solely on LLM client timeout; can hang indefinitely      |
| **State validation**              | **Missing** | No explicit state machine enforcer; implicit status checks only |
| **Checkpoint/resume**             | **Missing** | If process crashes mid-dimension, all progress lost             |
| **Fire-and-forget `void`**        | Partial     | Some missing `void` prefix on async calls                       |

### Event System

- **30+ event types** with strong typing (`LeaderThinkingData`, `AgentWorkingData`, etc.)
- Events persisted to `ResearchAgentActivity` table
- WebSocket gateway enforces **MAX_CONNECTIONS_PER_USER = 5** with oldest-disconnect policy
- No event storm protection beyond LLM backpressure

### Recommendations (Priority Order)

1. **P0**: Add 5-10 minute timeout per dimension task
2. **P1**: Implement state machine validator for explicit transition enforcement
3. **P1**: Add task checkpointing for crash recovery
4. **P2**: Add `void` prefix to all fire-and-forget promises
5. **P2**: Implement or remove `PLAN_READY` / `REVIEWING` states

## 3. Data Sourcing & Quality Pipeline (80/100)

### Data Source Architecture

**14 data sources** via Strategy pattern:

- Web, Academic, GitHub, HackerNews, RSS (stub)
- Local (RAG), Social X (Grok + web fallback)
- SemanticScholar, PubMed, FinanceAPI, WeatherAPI (connectors)
- FederalRegister, Congress, WhiteHouse (policy)

**Strengths**:

- `Promise.allSettled()` for parallel source execution
- Per-source error isolation (each in own try-catch)
- Circuit breaker integration (optional, via AI Kernel)
- Fallback chain: primary → secondary → WEB as last resort
- Domain diversity enforcement (prevents single-domain dominance)

**Gaps**:

| Gap                               | Severity | Impact                                                 |
| --------------------------------- | -------- | ------------------------------------------------------ |
| **No rate limiting**              | High     | `Promise.allSettled` fires all requests simultaneously |
| **No exponential backoff**        | High     | Fixed 30s timeout, no retry with backoff               |
| **Evidence sync queue ephemeral** | Medium   | In-memory queue lost on crash (max 1000, 5min retry)   |
| **Slow source blocks batch**      | Medium   | 25s source delays entire `allSettled`                  |

### 4-Layer Quality Pipeline

| Layer                  | Service                    | Type                             | Blocking                   |
| ---------------------- | -------------------------- | -------------------------------- | -------------------------- |
| L1: Content Rules      | `ReportQualityGateService` | Code-enforced (18 rules)         | Errors block, warnings log |
| L2: Claim Verification | `ClaimVerificationService` | AI-powered extractive            | Advisory (accuracy %)      |
| L3: Self-Consistency   | `SelfConsistencyService`   | Multi-path reasoning             | Advisory (agreement rate)  |
| L4: Critique-Refine    | `CritiqueRefineService`    | Iterative refinement (Reflexion) | Blocking (min improvement) |

**Quality Gate Rules**: 10 dimension-level + 8 report-level = **18 rules total**

- 9 auto-fixable (heading hierarchy, bold density, etc.)
- 9 require AI rewrite (language consistency, citation coverage, etc.)

### Skill Wiring Status

| Category           | Count | Percentage |
| ------------------ | ----- | ---------- |
| Total skill files  | 35    | 100%       |
| Wired into code    | ~22   | 63%        |
| Unused (dead code) | ~13   | 37%        |

**Unused skills** include: `cause-effect`, `comparison`, `competitive-analysis`, `critical-thinking`, `data-interpretation`, `deep-dive`, `future-projection`, `swot-analysis`, `synthesis`, `trend-analysis` — these are Leader-assignable via `section-writer.service.ts` `skillIdMapping`, but only when Leader explicitly selects them. Not dead code per se, but **demand-driven activation**.

### Recommendations

1. **P0**: Add token bucket rate limiter per data source
2. **P1**: Implement exponential backoff for retries (1s, 2s, 4s)
3. **P1**: Persist evidence sync queue to DB for crash recovery
4. **P2**: Add per-source adaptive timeout (not uniform 30s)
5. **P2**: Add token budget limit to Critique-Refine loop

## 7. Industry Best Practice Comparison

### vs. Benchmark: Multi-Agent Research Systems

| Practice          | Industry Standard                       | Topic Insights                     | Gap                        |
| ----------------- | --------------------------------------- | ---------------------------------- | -------------------------- |
| State Machine     | Explicit FSM with validated transitions | Implicit status checks             | Need explicit FSM          |
| Task Timeout      | Per-task timeout + circuit breaker      | Global deadlock only (60s)         | Need per-task timeout      |
| Checkpoint/Resume | Persistent checkpoints (Saga pattern)   | None — crash loses progress        | High priority gap          |
| Rate Limiting     | Token bucket per external API           | None                               | High priority gap          |
| Observability     | Distributed tracing (OpenTelemetry)     | In-memory event persistence        | Adequate for current scale |
| Quality Gates     | Automated blocking gates                | 4-layer (1 blocking + 3 advisory)  | Good                       |
| Error Budget      | SLO-based error budgets                 | Implicit (complete if any success) | Nice-to-have               |
